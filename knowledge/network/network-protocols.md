# 네트워크 프로토콜 기초 (OSI 7 Layer, TCP/UDP, HTTPS/TLS)

> **Context**: 모던 애플리케이션 개발 및 시스템 설계에 필수적인 표준 네트워크 아키텍처(OSI 7 Layer), 전송 계층의 대표 주자인 TCP/UDP의 실무 제어 관점 차이, 그리고 안전한 웹 보안 표준인 HTTPS와 TLS 핸드쉐이크의 기작을 정리한 지식 문서.

---

## 1. OSI 7 계층 (OSI 7 Layer)

국제표준화기구(ISO)에서 정의한 네트워크 통신 모델로, 상호 이질적인 네트워크 장비 간의 통신 호환성을 보장하기 위해 전체 네트워크 아키텍처를 7개 계층으로 추상화했다.

```
       [송신 (캡슐화)]                                [수신 (역캡슐화)]
┌─────────────────────────────┐                     ┌─────────────────────────────┐
│ 7. 응용 (Application)       │ ── 데이터 전송 ──►  │ 7. 응용 (Application)       │
├─────────────────────────────┤                     ├─────────────────────────────┤
│ 6. 표현 (Presentation)      │                     │ 6. 표현 (Presentation)      │
├─────────────────────────────┤                     ├─────────────────────────────┤
│ 5. 세션 (Session)           │                     │ 5. 세션 (Session)           │
├─────────────────────────────┤   [L4 Segment]      ├─────────────────────────────┤
│ 4. 전송 (Transport)         │ ──(헤더: 포트)────► │ 4. 전송 (Transport)         │
├─────────────────────────────┤   [L3 Packet]       ├─────────────────────────────┤
│ 3. 네트워크 (Network)      │ ──(헤더: IP 주소)─► │ 3. 네트워크 (Network)      │
├─────────────────────────────┤   [L2 Frame]        ├─────────────────────────────┤
│ 2. 데이터링크 (Data Link)   │ ──(헤더: MAC 주소)► │ 2. 데이터링크 (Data Link)   │
├─────────────────────────────┤   [L1 Bit Stream]   ├─────────────────────────────┤
│ 1. 물리 (Physical)          │ ──(전기/광 신호)──► │ 1. 물리 (Physical)          │
└─────────────────────────────┘                     └─────────────────────────────┘
```

### 1) 계층별 요약 및 프로토콜

| 계층 번호 | 계층명 | 역할 및 목적 | 주요 프로토콜 및 장비 |
|:---:|---|---|---|
| **L7** | **응용 계층 (Application)** | 사용자가 직접 접하는 인터페이스와 네트워크 서비스 제공 | HTTP, HTTPS, FTP, DNS, SMTP |
| **L6** | **표현 계층 (Presentation)** | 데이터 인코딩, 포맷 변환, 데이터 압축 및 암호화/복호화 수행 | SSL/TLS, ASCII, JPEG, MPEG |
| **L5** | **세션 계층 (Session)** | 통신 장비 간의 동기화, 커넥션 제어 및 연결 상태 유지 | SSH, NetBIOS, RPC |
| **L4** | **전송 계층 (Transport)** | 송신자와 수신자 간의 신뢰성 있고 실질적인 데이터 전송 흐름 제어 | **TCP, UDP** (포트 번호) |
| **L3** | **네트워크 계층 (Network)** | 다중 노드 네트워크를 거쳐 종단 간 패킷을 최적 경로로 전송 (라우팅) | **IP**, ARP, ICMP, 라우터 |
| **L2** | **데이터링크 계층 (Data Link)** | 물리 계층을 통해 신뢰성 있는 인접 노드 간 데이터 전송 보장 | 이더넷, MAC 주소, 스위치 |
| **L1** | **물리 계층 (Physical)** | 물리적 매체를 통해 가공되지 않은 이진 데이터(Bit)를 전기 신호로 전송 | 케이블, 허브, 리피터 |

### 2) 데이터의 캡슐화(Encapsulation) & 역캡슐화(Decapsulation)
- **캡슐화 (송신 측: 7계층 ➔ 1계층)**: 상위 계층에서 만들어진 데이터에 하위 계층으로 내려갈 때마다 해당 계층의 제어 정보(Header 및 데이터링크의 Trailer)를 덧씌워 물리적 비트 스트림으로 변환하는 과정.
- **역캡슐화 (수신 측: 1계층 ➔ 7계층)**: 수신한 물리 신호를 하위 계층부터 차례대로 읽어 해당 계층의 헤더 정보를 검증/해석한 후 분리하여 최종 유효 데이터를 상위 응용 프로그램에 넘겨주는 과정.

## 2. TCP vs UDP

### 1) 비교 특징

| 구분 | TCP (Transmission Control Protocol) | UDP (User Datagram Protocol) |
|---|---|---|
| **연결 지향성** | 연결형 (3-way handshake를 통한 가상 회선 구축) | 비연결형 |
| **신뢰성** | **매우 높음** (패킷 순서 보장, 재전송 제어) | **낮음** (순서 보장 불가, 유실 시 재처리 없음) |
| **속도 및 오버헤드** | 상대적으로 느림 (헤더가 무겁고 3-way, 흐름/혼잡 제어 수행) | 매우 빠름 (최소한의 헤더인 8바이트 구성, 제어 오버헤드 없음) |
| **흐름/혼잡 제어** | 제공 (Sliding Window, Slow Start 등) | 제공하지 않음 |
| **주요 사용 분야** | 웹 통신(HTTP), 이메일, 파일 전송(FTP) | 영상 스트리밍(RTP), 실시간 멀티플레이 게임, DNS |

---

### 2) 실시간 제어(Robotics, Drone, BLE)에서 UDP가 적합한 이유

실제 실시간 로보틱스 제어나 원격 차량 조작, 혹은 멀티플레이어 FPS 슈팅 게임 등의 물리적 시나리오에서는 **신뢰성 높은 TCP보다 UDP가 압도적으로 권장**된다.

* **TCP의 치명적 한계: HOL (Head-of-Line) Blocking**
  * TCP는 수신부에서 패킷 순서가 어긋나거나 유실(Loss)이 일어나면, 깨진 패킷이 재전송(`ACK/NACK` 메커니즘)을 통해 도착할 때까지 그 뒤에 정상적으로 도착한 최신 패킷들을 상위 애플리케이션으로 올려 보내지 않고 홀딩(버퍼링)한다.
  * 이로 인해 일시적인 무선 감쇄 현상이 일어났을 때 전송 지연(Latency Spike)이 누적되어 시스템 무응답(Lag)이 초래된다.
* **UDP의 실시간 대처 원리**
  * 실시간 제어 장치에서는 **"5초 전에 유실된 좌표 데이터"는 아무런 쓸모가 없으며, 오직 "지금 이 순간 도착한 최신 좌표 데이터"만이 유효**하다.
  * UDP는 패킷 유실을 완벽히 무시하고 뒤이어 들어오는 패킷을 가로막지 않고 즉시 런타임에 주입한다. 
  * 따라서 패킷이 다소 깨지더라도 지연을 발생시키지 않고 최신 제어 명령을 실시간(Real-time)으로 갱신해 복구할 수 있다.

## 3. HTTP vs HTTPS 및 TLS Handshake

### 1) 보안 차이점
- **HTTP (HyperText Transfer Protocol)**: 평문 텍스트 형식으로 송수신하므로, 중간에서 스니핑(Sniffing)이나 패킷 도청이 일어났을 때 중요한 개인정보나 쿠키 토큰 등이 날것 그대로 노출된다.
- **HTTPS (HTTP Secure)**: HTTP 하부에 **TLS(Transport Layer Security)** 암호화 계층을 두어 통신 페이로드를 가로채더라도 해독을 불가능하게 만든다. 추가로 신뢰할 수 있는 기관(CA)의 디지털 인증서로 서버의 신원을 증명하여 피싱 사이트 차단 및 신원 검증을 보장한다.

---

### 2) TLS Handshake (TLS 1.2 기준 상세 기작)

HTTPS 연결 수립 시, 클라이언트와 서버가 암호화 키를 안전하게 공유하기 위해 거치는 악수(Handshake) 시퀀스다.

```
[Client]                                                        [Server]
   │                                                               │
   │ 1) Client Hello (지원 cipher suite, 무작위 난수 A 송신)        │
   ├──────────────────────────────────────────────────────────────►│
   │                                                               │
   │ 2) Server Hello (Cipher suite 결정, 무작위 난수 B 송신)        │
   │ 3) Server Certificate (CA 서명 인증서, 서버 공개키 포함)      │
   │◄──────────────────────────────────────────────────────────────┤
   │                                                               │
   │ [Client: CA 서명 검증 및 Pre-master Secret 생성]              │
   │ 4) Client Key Exchange (서버 공개키로 Pre-master Secret 암호화)│
   ├──────────────────────────────────────────────────────────────►│
   │                                                               │
   │                               [Server: 개인키로 복호화 ➔ 키 획득]
   │                                                               │
   │ [Client/Server: 난수 A, B, Pre-master로 Session Key 생성]      │
   │                                                               │
   │ 5) Change Cipher Spec & Finished (핸드쉐이크 완료 안내)        │
   ├──────────────────────────────────────────────────────────────►│
   │ 6) Change Cipher Spec & Finished (핸드쉐이크 완료 안내)        │
   │◄──────────────────────────────────────────────────────────────┤
   │                                                               │
   │ 7) Application Data (Session Key인 대칭키로 페이로드 암호화 통신)│
   │◄─────────────────────────────────────────────────────────────►│
```

#### ① Client Hello (연결 요청)
- 클라이언트가 서버에 접속을 시도하며 다음 데이터를 전달한다.
  - 클라이언트가 생성한 임의의 무작위 난수 (Random bytes)
  - 지원 가능한 대칭/비대칭 암호 알고리즘 목록 (Cipher Suites)
  - TLS 버전 프로토콜

#### ② Server Hello, Certificate (응답 및 신원 증명)
- 서버는 클라이언트의 Cipher Suites 중 사용할 알고리즘을 선택해 통보한다.
  - 서버가 생성한 임의의 무작위 난수 (Random bytes)
  - **서버 인증서 (Certificate)**: CA의 서명이 있고, **서버의 비대칭형 공개키(Public Key)**가 포함되어 있다.

#### ③ 인증서 검증 (Certificate Verification)
- 클라이언트는 브라우저/OS 내장 저장소에 탑재된 CA의 공개키를 꺼내 서버가 보낸 인증서의 서명을 복호화 및 검증한다. 검증에 성공하면 이 서버의 신원이 확실하며, 포함된 공개키도 신뢰할 수 있다고 판단한다.

#### ④ Key Exchange (Pre-master Secret 암호화 및 전송)
- 클라이언트는 세 번째 무작위 바이트 조합인 **Pre-master Secret(실제 세션 대칭키의 원천)**을 직접 생성한다.
- 그 후, 검증된 서버의 **공개키(Public Key)**로 이 Pre-master Secret을 암호화하여 서버로 전송한다.

#### ⑤ 복호화 및 대칭키 유도 (Decryption & Session Key Derivation)
- 서버는 오직 자신만이 소유한 **비대칭형 개인키(Private Key)**로 전송받은 값을 복호화하여 Pre-master Secret을 안전하게 획득한다. (이 과정 덕분에 네트워크를 스니핑하는 제3자는 개인키가 없어 획득할 수 없음).
- 클라이언트와 서버는 각자 확보한 `난수 A`, `난수 B`, `Pre-master Secret` 조합을 연산하여 통신 암호화에 쓸 **최종 세션 대칭키(Session Key)**를 동치 유도한다.

#### ⑥ Change Cipher Spec & Finished (통신 전환)
- 양측은 "이제부터 대칭 세션키를 사용하여 데이터를 송수신하겠다"고 Finished 메시지를 교환하며 핸드쉐이크를 종료한다.
- 이후 본문 통신(Application Data)은 가볍고 신속한 **대칭키** 암호화 방식으로 전송된다.

---

### 3) 대칭키 vs 비대칭키 하이브리드 설계 방식의 가치
- **비대칭키(공개키/개인키)**: 안전한 키 공유가 가능하지만 연산 처리가 매우 느려, 대용량 웹 데이터를 실시간으로 통신하기엔 하드웨어 오버헤드가 극심하다.
- **대칭키**: 암복호화 속도가 매우 빠르지만 처음에 키를 네트워크로 그냥 보내면 해커에게 유출된다는 심각한 키 분배 문제가 있다.
- **하이브리드 해결책**: HTTPS는 **핸드쉐이크 과정에서 보안성이 우수한 비대칭키를 사용해 대칭 세션키(Session Key)를 안전하게 교환하고, 핸드쉐이크가 완료된 실제 통신 데이터 암호화에는 빠른 대칭키를 사용하는** 구조적 결합을 통해 속도와 보안을 동시에 만족시켰다.

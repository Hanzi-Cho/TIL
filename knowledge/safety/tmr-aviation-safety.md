# 항공 안전 시스템 — Triple Modular Redundancy (TMR)

> 비행기가 반드시 실시간성을 보장하는 세 가지 설계 조합

## TMR 구조

```
센서 입력
    │
    ├──→ 컴퓨터 A ──┐
    ├──→ 컴퓨터 B ──┼──→ 투표기(Voter) ──→ 실제 제어
    └──→ 컴퓨터 C ──┘

         2/3 다수결로 결정
```

세 독립 컴퓨터가 동일 입력을 받아 각각 계산.  
2개 이상이 동의한 값을 채택.  
하나가 오작동해도 나머지 둘이 올바른 값 유지.

---

## 세 가지 핵심 조합

### ① 하드웨어 중복 (Redundancy)
```
단일 장애점(Single Point of Failure) 제거
컴퓨터 A가 죽어도 B, C가 계속 동작
```

### ② RTOS — 결정론적 실행 (Determinism)
```
인터럽트 응답 시간: 마이크로초 단위 보장
스케줄러: 우선순위 역전(Priority Inversion) 없음
→ 언제 실행해도 같은 시간 안에 반드시 응답
```

### ③ Watchdog + 투표기 (Voting)
```
각 컴퓨터 → 주기적 heartbeat 전송
무응답 시 → 해당 컴퓨터 격리, 나머지 둘로 운영
투표기 → 다수결로 최종 값 결정
→ 한 컴퓨터가 잘못된 값 내도 무력화
```

---

## 항공 vs 자동차 비교

| | 항공 (DO-178C) | 자동차 (ISO 26262) |
|---|---|---|
| 중복 방식 | TMR (3중화) | 이중화 (Dual ECU) |
| 실시간성 | RTOS 결정론적 실행 | RTOS + CAN Bus 우선순위 |
| 최고 안전 등급 | DAL A | ASIL D |
| 결함률 요구 | 극도로 낮음 | 10^-8/시간 이하 |

자동차 ASIL D 기능(에어백, 조향)도 이중 ECU 교차 검증을 쓰는 경우 많음.  
항공만큼 3중화까지는 잘 안 가지만 방향은 같음.

---

## 핵심 인사이트

> **중복(Redundancy)** — 하나가 죽어도 계속 동작  
> **결정론(Determinism)** — 언제나 같은 시간 안에 응답  
> **투표(Voting)** — 잘못된 값이 실제 제어에 반영되지 않도록  
>
> 이 세 가지가 합쳐져야 "반드시 동작함"을 보장할 수 있다.  
> 소프트웨어 혼자서는 절대 달성 불가 — 하드웨어 설계와 함께 가야 한다.

## 관련 개념
- TMR (Triple Modular Redundancy)
- Priority Inversion (우선순위 역전) — RTOS에서 방지 필수
- DO-178C (항공 소프트웨어 안전 표준)
- ISO 26262 ASIL D
- CAN Bus 우선순위 중재
- CarWatchdog (Android IVI에서의 유사 패턴)

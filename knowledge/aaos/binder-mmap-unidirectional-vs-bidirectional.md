# Binder mmap 단방향 vs Binder IPC 양방향

> "mmap이 단방향인데 Binder IPC는 왜 양방향인가?"

## 핵심 구분

mmap 자체는 단방향 전달 메커니즘이다.  
Binder IPC가 양방향인 건 mmap을 **요청/응답으로 두 번** 쓰기 때문이다.

```
프로세스 A (클라이언트)          커널 Binder 드라이버
      │                               │
      │──── 요청 데이터 write ────────▶│
      │         (mmap, 1회 복사)       │──▶ 프로세스 B로 전달
      │                               │
      │◀─── 응답 데이터 read ──────────│
      │         (mmap, 1회 복사)       │◀── 프로세스 B가 응답
```

```
양방향 통신 = 단방향 mmap 전달 × 2회
             (요청 1회 + 응답 1회)
```

---

## 소켓과 비교

```
소켓 왕복:
A → send buffer → 커널 → recv buffer → B   (복사 1회)
B → send buffer → 커널 → recv buffer → A   (복사 1회)
총: 버퍼 4개 경유, 복사 2회

Binder 왕복:
A → 커널 mmap → B   (복사 1회)
B → 커널 mmap → A   (복사 1회)
총: 중간 버퍼 없음, 복사 2회
```

왕복 기준 복사 횟수는 동일하지만 **중간 버퍼를 거치지 않아서** 소켓보다 빠르다.  
"Binder가 1회 복사"라는 말은 **단방향 전달 1회 기준**이다.

---

## 정리

| | mmap 1회 | Binder IPC 왕복 |
|---|---|---|
| 복사 횟수 | 1회 | 2회 (요청+응답) |
| 방향 | 단방향 | 양방향 |
| 중간 버퍼 | 없음 | 없음 |

> mmap = 단방향 전달 메커니즘  
> Binder IPC = mmap을 요청/응답으로 두 번 써서 양방향 통신

# 온디바이스 AI 탑재 아키텍처 및 모델 비교

> **React Native + JSI 기반 고성능 온디바이스 AI 탑재 방식과 오픈 가중치 SLM 모델들의 핵심 기술 스펙 비교 분석**

---

## 1. 온디바이스 AI 탑재 방식의 구조적 분류

안드로이드 및 모바일 환경에서 인공지능 모델을 온디바이스 단독 구동하기 위해 설계할 수 있는 아키텍처는 크게 두 가지 수준으로 나뉩니다.

### A. System-level (시스템 내장형)
*   **대표 모델/인프라**: 구글 Gemini Nano, Android AICore.
*   **원리**: 모바일 운영체제(OS) 레이어 단에서 기본 탑재된 공용 가중치 모델과 NPU 가속 드라이버를 직접 통제합니다.
*   **장점**: 모델이 시스템 프로세스 영역에 상주하므로, 개별 앱 프로세스의 메모리(RAM) 부담이 거의 없고 OOM(Out Of Memory) 리스크가 최소화됩니다.
*   **단점**: 구글 픽셀 등 최신 플래그십 기기에서만 선택적으로 활성화되며, iOS 등 타 플랫폼 이식이나 커스텀 모델 서빙이 불가능한 플랫폼 종속성을 띱니다.

### B. App-level (앱 내포형)
*   **대표 모델/인프라**: Gemma, Llama, Qwen 등 오픈 가중치 + llama.cpp, ONNX Runtime 추론 엔진.
*   **원리**: 개발자가 직접 양자화된 모델 가중치 파일(`.gguf`, `.bin`)과 네이티브 C++ 추론 라이브러리를 앱 빌드 패키지에 번들링하여 함께 배포합니다.
*   **장점**: 안드로이드와 iOS 크로스 플랫폼 대응이 즉각 가능하고, 오프라인 환경에서 도메인에 특화된 가중치 파인튜닝 모델을 독립적으로 서빙할 수 있습니다.
*   **단점**: 앱 설치 패키지 용량(APK/IPA)이 최소 수백 MB에서 GB 단위로 폭증하며, 실행 시 앱 프로세스의 힙 메모리(RAM) 영역을 최소 1.5GB ~ 3GB 직접 점유하여 백그라운드 킬(Kill) 및 OOM에 취약해집니다.

## 2. 온디바이스 SLM(소형 언어 모델) 대안군 분석

App-level 구조 채택 시, 모바일 런타임 환경에서 서빙하기 유용한 INT4 양자화 기반 오픈 SLM 모델들의 특성을 분석하여 추천 시나리오를 정의합니다.

| 모델명 | 체급 (INT4 기준 용량) | JSI 아키텍처 관점의 핵심 강점 및 추천 시나리오 |
| :--- | :--- | :--- |
| **Gemma 2B / 4B** | ~1.5 GB 이상 | 구글의 대표적 경량 모델. 이미지/오디오 등 멀티모달(Multimodal) 입출력 전처리가 로컬에서 복합 연동되어야 하는 상황에 강력함. |
| **Llama 3.2 1B / 3B** | ~1.2 GB 이상 | 너비 중심(Width)의 아키텍처 최적화. 모바일 GPU 연산 효율이 뛰어나 초기 토큰 생성 시간(Time to First Token)이 가장 빠름. 모바일 챗봇 스택에 적합. |
| **Phi-4 Mini 3.8B** | ~2.2 GB | 마이크로소프트의 논리 추론 특화 모델. 고품질로 정제된 데이터 학습을 거쳐 동급 체급 중 벤치마크 결과가 최상위임. 복잡한 계산이나 정밀 지시 이행에 적합. |
| **Qwen 2.5 1.5B / 3B** | ~1.1 GB 이상 | 구조화된 출력(Structured Output, 예: JSON 포맷) 출력 성공률이 최상위임. 지시 이행률이 높아 아웃풋을 JS 객체로 파싱하는 JSI 브릿지 변환 시 포맷 깨짐 예외 방지에 가장 강함. |

## 3. 브릿지 병목과 온디바이스 AI 연동 트레이드오프 해결

### A. 문제 상황 (Why)
*   기존 React Native Bridge(JSON 직렬화 메시지 큐 방식) 환경에서 매 초 수십 번씩 갱신되는 원본 음성 바이트 스트림(raw PCM)이나 대용량 카메라 프레임 이미지를 AI 모델로 넘길 경우, 브릿지 병목으로 인해 화면 전체가 멈추는 UI 프리징과 심각한 오디오 버벅임이 발생함.

### B. 해결 방안 (How)
*   **JSI(JavaScript Interface) C++ Direct Access**: JSI 기반 고성능 네이티브 모듈 영역을 메인 파이프라인 채널로 변경.
*   C++ 단에서 하드웨어 버퍼 포인터에 직접 접근하고, 데이터 원본은 복사 없이(Zero-copy) C++ 메모리 풀에 상주한 상태에서 추론 엔진(Whisper.cpp, Llama.cpp)이 직접 긁어가도록 아키텍처를 연계 설계함.

### C. 결과 (Result)
*   자바스크립트-네이티브 스레드 간 데이터 중복 복사 비용을 영(Zero)으로 낮춤으로써 모바일 단독 실시간 추론 시스템 구동을 위한 토대 마련.

## 4. 서비스 목적별 온디바이스 AI 아키텍처 설계 구조

```text
[시나리오 1: 로컬 오디오/음성 인식 위주]
하드웨어 입력 -> [JSI C++ Direct Buffer] -> [전용 ASR (Whisper.cpp)] -> JS 레이어 동기식 반환

[시나리오 2: 로컬 데이터 전처리 및 에이전트 구동 위주]
사용자 입력 -> [JSI Binding] -> [Qwen 2.5 C++ Engine] -> 로컬 JSON 포맷 추출 -> 네이티브 함수 실행
```

*   **Action Item**: 선택된 오픈 가중치 모델을 INT4 수준으로 경량 양자화한 뒤, MediaPipe LLM Inference API 또는 llama.cpp를 C++ 네이티브 모듈에 이식하고 JSI HostObject로 래핑하는 프로토타입 구현 진행 예정.

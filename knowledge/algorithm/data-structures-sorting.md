# 핵심 자료구조 및 정렬 알고리즘 (Data Structures and Sorting Algorithms)

> **Context**: 소프트웨어 개발의 가장 기본이 되는 데이터 가공/저장 메커니즘인 선형 자료구조(배열, 연결 리스트, 스택, 큐)의 물리적 메모리 배치 특성과 각 자료구조에 최적화된 연산 성능, 그리고 대표적인 5대 정렬 알고리즘의 시간/공간 복잡도와 정렬 특성을 학습하고 정리한 지식 문서.

---

## 1. 선형 자료구조 (Linear Data Structures)

### 1) Array vs Linked List (배열 vs 연결 리스트)

데이터를 메모리에 배치하는 물리적인 아키텍처에 따라 조회 및 삽입/삭제 성능의 극단적인 트레이드 오프가 발생한다.

```
[Array: 연속된 메모리 공간 배치]
┌───────┬───────┬───────┬───────┐
│ Val A │ Val B │ Val C │ Val D │  (Index 0 ➔ 1 ➔ 2 ➔ 3 연속 접근)
└───────┴───────┴───────┴───────┘
  1000    1004    1008    1012   (물리 주소: Base + Index * Size)

[Linked List: 분산된 메모리 공간 배치]
┌──────────────┐      ┌──────────────┐      ┌──────────────┐
│ Val A ┆ Next ├─────►│ Val B ┆ Next ├─────►│ Val C ┆ Next │
└──────────────┘      └──────────────┘      └──────────────┘
  주소: 0x401F          주소: 0x90A2          주소: 0x1102
```

#### ① 배열 (Array)
- **물리 구조**: 메모리의 **연속된 공간(Contiguous Space)**에 데이터를 배치한다.
- **주소 연산**: 시작 주소(Base Address)와 인덱스를 이용해 목적지 주소를 수학적으로 즉시 계산한다.
  - $\text{Address(i)} = \text{Base} + i \times \text{DataSize}$
  - 이에 따라 임의의 인덱스에 **$O(1)$의 상수 시간으로 즉각 접근(Random Access)**할 수 있다. (캐시 지역성 우수)
- **한계**: 크기가 정적(Fixed Size)으로 결정되므로 늘릴 수 없으며, 중간에 데이터를 삽입/삭제할 경우 기존 데이터들을 한 칸씩 뒤로 밀거나 앞으로 당겨야 하므로 **$O(n)$의 데이터 이동 비용**이 발생한다.

#### ② 연결 리스트 (Linked List)
- **물리 구조**: 메모리 상의 **비연속적인 임의의 공간**에 데이터를 흩어놓고, 각 데이터 노드가 다음 노드의 메모리 주소를 가리키는 **포인터(Pointer)**로 연결한다.
- **주소 연산**: 임의의 노드로 바로 점프할 수 없으며, 무조건 머리 노드(Head)부터 링크 포인터를 타고 순차 탐색(Sequential Access)해야 하므로 특정 위치 접근에 **$O(n)$의 비용**이 소요된다.
- **장점**: 크기가 동적으로 조절(Dynamic Size)되며, 노드의 추가/삭제 시 주변 노드의 포인터 링크 주소 값만 바꾸면 되므로 **$O(1)$의 상수 시간으로 빠르게 삽입/삭제**가 가능하다.

#### 연산 복잡도 대조표

| 작업 (Complexity) | 배열 (Array) | 연결 리스트 (Linked List) |
|---|---|---|
| **임의 조회 (Access)** | **$O(1)$** | $O(n)$ |
| **순차 탐색 (Search)** | $O(n)$ | $O(n)$ |
| **맨 앞 삽입/삭제** | $O(n)$ (전체 엘리먼트 시프트 발생) | **$O(1)$** (Head 포인터만 교체) |
| **맨 뒤 삽입/삭제** | **$O(1)$** (공간 여유 있을 시) | $O(n)$ (단, Tail 포인터를 들고 있다면 $O(1)$) |
| **중간 삽입/삭제** | $O(n)$ (탐색 후 시프트 비용 발생) | $O(n)$ (탐색 $O(n)$ 후 삽입/삭제 자체는 $O(1)$) |
| **메모리 오버헤드** | 없음 (순수 데이터만 저장) | 포인터 저장용 추가 메모리 필요 |

### 2) Stack vs Queue (스택 vs 큐)

특정 순서로만 데이터를 처리하도록 제한한 선형 자료구조이다.

```
[Stack: LIFO 구조]                      [Queue: FIFO 구조]
  입구/출구 (맨 뒤)                       입구 (맨 뒤)           출구 (맨 앞)
 ┌───────────────┐                      ┌───────────────────────────┐
 ◄── push / pop  │                      ◄── enqueue         dequeue ◄──
 └───────────────┘                      └───────────────────────────┘
```

#### ① 스택 (Stack)
- **원리**: **LIFO (Last In First Out, 후입선출)** 구조. 가장 마지막에 삽입된 데이터가 가장 먼저 제거된다.
- **물리 구현 최적화 (Array가 유리)**:
  - 스택은 데이터의 삽입(`push`)과 삭제(`pop`)가 **항상 리스트의 맨 끝(Top)**에서만 일어난다.
  - 배열(Array)은 맨 끝 데이터를 조작할 때 인덱스를 직접 가리키므로 시프트 오버헤드가 전혀 없어 $O(1)$로 작동한다.
  - 또한, 연속된 메모리에 상주하므로 CPU 캐시 히트율(Locality)이 높아 스택 구현에 배열이 가장 효율적이다.
- **실무 사례**:
  - 함수 호출의 복귀 주소 및 지역변수를 저장하는 **시스템 호출 스택 (Call Stack)**
  - 웹 브라우저의 **뒤로 가기/앞으로 가기** 히스토리 관리
  - Android 컴포넌트 관리의 **백스택 (BackStack)** 구조

#### ② 큐 (Queue)
- **원리**: **FIFO (First In First Out, 선입선출)** 구조. 가장 먼저 삽입된 데이터가 가장 먼저 제거된다.
- **물리 구현 최적화 (Linked List가 유리)**:
  - 큐는 **맨 뒤(Rear)로 데이터를 넣고(`enqueue`), 맨 앞(Front)에서 데이터를 꺼내는(`dequeue`)** 양방향 조작 구조다.
  - 만약 일반 배열로 구현한다면 맨 앞 원소를 뺄 때마다 뒤에 있는 모든 데이터를 앞으로 한 칸씩 땡겨야 하므로 `dequeue` 시 $O(n)$의 지연이 일어난다. (Circular Queue로 우회할 수 있으나 크기 제약 존재)
  - 연결 리스트로 구현 시 Head 포인터 조작을 통해 **$O(1)$의 속도로 맨 앞 데이터를 즉시 제거**할 수 있어 큐 구현에 최적이다.
- **실무 사례**:
  - Android 루퍼의 비동기 메시지 수신 버퍼인 **MessageQueue**
  - 프린터의 인쇄 대기열, CPU의 프로세스 스케줄링 준비 큐 (Ready Queue)
  - 그래프 탐색 알고리즘인 **BFS (Breadth-First Search, 너비 우선 탐색)**

## 2. 정렬 알고리즘 (Sorting Algorithms)

정렬 알고리즘은 성능(시간/공간 복잡도)과 **안정성(Stability)** 및 **제자리 정렬 여부(In-place)**에 따라 선택한다.
* *안정 정렬(Stable Sort)*: 정렬 후에도 값이 같은 원소들의 상대적인 순서가 유지되는 정렬.
* *제자리 정렬(In-place Sort)*: 입력 배열 이외에 추가적인 메모리 공간을 거의 소모하지 않고 정렬을 수행하는 방식.

### 1) 5대 정렬 알고리즘 복잡도 종합

| 알고리즘 (Sort) | 시간 복잡도 (최선) | 시간 복잡도 (평균) | 시간 복잡도 (최악) | 공간 복잡도 | 안정성 (Stable) | 특징 |
|---|:---:|:---:|:---:|:---:|:---:|---|
| **버블 정렬 (Bubble)** | $O(n^2)$ | $O(n^2)$ | $O(n^2)$ | $O(1)$ | Stable | 인접한 두 원소를 비교하며 스왑. 구현이 가장 단순하나 성능이 매우 낮음. |
| **선택 정렬 (Selection)** | $O(n^2)$ | $O(n^2)$ | $O(n^2)$ | $O(1)$ | Unstable | 매 회차 최솟값을 찾아 맨 앞으로 교환. 비교 횟수는 고정이나 스왑 횟수가 적음. |
| **삽입 정렬 (Insertion)** | **$O(n)$** | $O(n^2)$ | $O(n^2)$ | $O(1)$ | Stable | 앞의 정렬된 부분에 새 원소를 올바른 위치에 삽입. 거의 정렬된 배열에 특화. |
| **퀵 정렬 (Quick)** | $O(n \log n)$ | **$O(n \log n)$** | $O(n^2)$ | $O(\log n)$ | Unstable | 피벗(Pivot) 기준으로 파티셔닝 수행 후 분할 정복. 실무 평균 속도가 가장 빠름. |
| **병합 정렬 (Merge)** | $O(n \log n)$ | **$O(n \log n)$** | **$O(n \log n)$** | $O(n)$ | Stable | 분할 후 병합하며 정렬. 항상 일정한 성능을 보장하나 추가 메모리가 필수적임. |

---

### 2) 알고리즘별 핵심 기작

#### ① 버블 정렬 (Bubble Sort)
- **원리**: 서로 이웃한 두 원소를 비교하여 크기 순서가 맞지 않으면 스왑(Swap)하는 과정을 첫 번째 원소부터 마지막 원소까지 반복한다. 1회전이 끝나면 가장 큰 원소가 맨 뒤로 배치(거품이 올라오듯)된다.
- **복잡도**: 모든 경우에 2중 루프를 돌기 때문에 항상 $O(n^2)$의 시간 복잡도를 가진다.

#### ② 선택 정렬 (Selection Sort)
- **원리**: 주어진 배열에서 최솟값을 찾아 선두의 데이터와 교환한다. 그 다음 선두를 제외한 나머지 영역에서 최솟값을 찾아 두 번째 데이터와 교환하는 과정을 반복한다.
- **복잡도**: 최솟값 비교 연산이 고정되어 있어 최선, 평균, 최악 모두 $O(n^2)$이다.

#### ③ 삽입 정렬 (Insertion Sort)
- **원리**: 두 번째 원소부터 시작하여 그 앞의 이미 정렬된 부분 배열 사이에서 본인의 자리를 찾아 삽입한다.
- **특이성**: 배열이 **이미 완전히 또는 거의 정렬된 상태라면 비교 즉시 루프가 끝나기 때문에 $O(n)$의 극단적으로 빠른 속도**로 정렬이 종료된다.

#### ④ 퀵 정렬 (Quick Sort)
- **원리**: 배열에서 기준점인 **피벗(Pivot)**을 하나 선택한다. 피벗보다 작은 원소들은 왼쪽으로, 큰 원소들은 오른쪽으로 이동시킨 뒤(Partitioning), 좌우 하위 배열을 각각 재귀적으로 정렬(Divide and Conquer)한다.
- **한계와 대안**: 피벗이 계속 최댓값이나 최솟값으로 잡히는 불균형 상황(예: 이미 정렬된 배열에서 맨 앞을 피벗으로 선택할 때) 최악의 복잡도 **$O(n^2)$**이 초래된다. 이를 방지하기 위해 피벗을 무작위로 고르거나 세 값의 중앙값(Median of Three)을 고르는 기법을 적용한다.

#### ⑤ 병합 정렬 (Merge Sort)
- **원리**: 배열의 크기가 1이 될 때까지 반으로 쪼갠 뒤, 쪼개진 부분 배열들을 정렬해가며 하나로 합쳐 나가는 분할 정복 방식.
- **장단점**: 피벗 설정 등의 변수 없이 항상 절반으로 쪼개기 때문에 **최악의 경우에도 무조건 $O(n \log n)$을 보장**하며, 순서 보존이 필요한 Stable Sort다. 단, 합치는 과정에서 데이터를 임시로 저장할 동일 크기의 **추가 임시 배열($O(n)$ 공간)**이 필요하다는 물리적 단점이 존재한다.

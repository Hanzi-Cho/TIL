# AI 시대의 React 및 프론트엔드 아키텍처 핵심 면접 질문 11선

이 문서는 실무 5년차 이상 시니어 레벨의 아키텍처 감수성, 비동기 통제 능력, 성능 디버깅 및 AI 기반 개발 패러다임에서의 역할 정의 능력을 검증하기 위해 정리한 면접 질문 리스트입니다.

---

### Q1. React의 렌더링 파이프라인과 Concurrent Rendering
* React의 렌더링 메커니즘을 Render Phase, Commit Phase, Reconciliation(조화) 및 Fiber 아키텍처 관점에서 설명하세요.
* **꼬리 질문**: React 16 이후 이전의 스택 기반 재조정자 대신 Fiber 구조로 전면 전환된 컴퓨터 아키텍처적 배경은 무엇이며, React 18의 Concurrent Rendering(동시성 렌더링)이 이를 어떻게 활용하나요?

### Q2. 프론트엔드-백엔드 간 API 설계 계약(API Contract)
* API 규격 설계 시 프론트엔드 개발자 관점에서 고려해야 하는 데이터 핸들링 요건(Pagination 전략, Sorting/Filtering, Partial Response, API Versioning 등)을 설명하세요.
* **꼬리 질문**: 전통적인 REST 아키텍처가 지닌 오버패칭(Over-fetching) 및 언더패칭(Under-fetching)의 한계는 무엇이며, 어떤 상황에서 GraphQL이나 tRPC 같은 대체 기술을 채택해야 합니까?

### Q3. 상태(State)의 역할 분리와 서버 상태(Server State) 모델
* 모던 프론트엔드 프로젝트에서 다루는 상태의 성격(Local, Global, Server State, Cache)을 명확히 구분하여 설계 방침을 기술하세요.
* **꼬리 질문**: 왜 최근 프로젝트에서 Redux/Recoil 같은 전역 상태 관리 도구의 비중이 줄어들고, React Query(TanStack Query)나 SWR 같은 서버 상태 관리 라이브러리가 그 역할을 대체하게 되었는지 아키텍처적 변화를 설명하세요.

### Q4. AI 생성 코드에 대한 검증 및 코드 리뷰 사양
* AI 코딩 어시스턴트가 자동으로 생성해 온 300줄 이상의 대규모 복잡 소스 코드를 프로젝트에 머지하기 전, 코드 리뷰어로서 검증해야 하는 핵심적인 품질 체크리스트(의존성 방향, Side Effect, 예외 처리, Race Condition, 성능 잠재 리스크 등)를 기술하세요.

### Q5. 비동기 처리의 경합 상태(Race Condition)와 부작용 제어
* React 비동기 데이터 페칭 중 발생할 수 있는 Race Condition(예: 연속적인 API 요청 후 뒤늦게 도착한 응답이 컴포넌트 화면을 잘못 덮어쓰는 상황)의 구체적인 시나리오를 설명하세요.
* **꼬리 질문**: 이를 방지하기 위한 AbortController API의 원리와 이를 응용한 클린업 렌더링 흐름을 설명하세요.

### Q6. 레이어드 아키텍처 기반의 에러 핸들링 정책
* 클라이언트 아키텍처의 각 계층(Network Layer -> API Layer -> Repository -> Custom Hook -> Component)에서 감지해야 할 에러의 성격과 적절한 에러 전파/처리 기법을 정의하세요.
* **꼬리 질문**: React Error Boundary는 선언적으로 런타임 에러를 포착하지만, 잡아낼 수 없는 예외 상황들(비동기 에러, 이벤트 핸들러 내부 에러 등)은 무엇이며 이를 어떻게 우회 처리하나요?

### Q7. 대규모 어플리케이션의 의존성 흐름 및 순환 의존성 제어
* 레이어드 컴포넌트 설계 시 발생하기 쉬운 순환 의존성(Circular Dependency)의 원인과 이를 정적 분석 및 아키텍처 규칙으로 차단하는 방법을 제시하세요.
* **꼬리 질문**: 도메인 기능 기반 구조(Feature-driven Architecture)와 전통적인 계층 기반 구조(Layer-based Architecture)의 아키텍처적 장단점을 확장성 관점에서 논하세요.

### Q8. 런타임 성능 프로파일링 및 렌더링 최적화 디버깅
* 특정 복잡 페이지의 성능 저하(부하, 프레임 드랍, 입력 지연)가 보고되었을 때, 이를 추적하기 위해 진단해야 할 지표들(Network 지연, Bundle Size, 무의미한 부모 리렌더링, Layout Shift, Memory Leak 등)과 디버깅 순서를 제시하세요.
* **꼬리 질문**: React DevTools Profiler의 Flamegraph 및 Ranked 차트를 분석하여 렌더링 병목을 포착하는 기법을 설명하세요.

### Q9. 대규모 React 프로젝트의 도메인 모듈화와 디렉토리 설계
* 다중 비즈니스 도메인(User, Order, Payment, Notification 등)이 결합된 대형 프로젝트에서 모듈 간 결합도를 낮추기 위한 디렉토리 구조 및 경계 설계 방안을 설명하세요.
* **꼬리 질문**: Shared(공통 모듈), Feature(도메인 특화 기능), Core(어플리케이션 생명주기 제어), Common(유틸리티) 간의 참조 규칙(Dependency Rule)을 정의하세요.

### Q10. 신규 기능 설계 및 개발 접근 프로세스
* "실시간 알림 및 인터랙션 기능"을 백지 상태에서 개발한다고 할 때, 설계를 조율하고 비즈니스 신뢰성을 높이기 위한 단계적 개발 흐름(요구사항 분석 -> API 규격 합의 -> 로컬 상태 설계 -> 로딩/스켈레톤 UX -> 낙관적 업데이트(Optimistic Update) -> 예외/폴백 처리 -> 실시간 로깅 및 센트리 연동)을 설명하세요.

### Q11. AI 자동화 개발 패러다임에서의 5년차 이상 시니어 프론트엔드 개발자의 역할
* AI 도구가 보편화되어 단순 코드 구현 속도의 가치가 희석되는 개발 환경에서, 5년차 이상의 시니어 개발자가 집중해야 하는 본질적인 가치(아키텍처 경계 설계, 데이터 흐름 제어, 에러 복구 모니터링 정책 수립, 코드의 운영 및 유지보수 가능성 검증 등)에 대해 기술 철학을 밝히세요.

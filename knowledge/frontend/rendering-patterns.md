# 웹 렌더링 패턴 가이드: SPA, SSR, SSG

이 문서는 모던 웹 어플리케이션의 핵심 아키텍처 패턴인 SPA, SSR, SSG의 기본 메커니즘과 동작상의 장단점, 그리고 성능 핵심 지표(TTV, TTI, Hydration)를 비교 정리한 지식 문서입니다.

---

## 1. SPA (Single Page Application)

SPA는 단 하나의 HTML 파일만 브라우저에 내려받은 뒤, 화면 전환 시 서버로부터 새 HTML을 다시 받아오지 않고 **자바스크립트가 동적으로 브라우저 DOM 트리를 통째로 갈아 끼워 그리는(Client-Side Rendering) 아키텍처**입니다.

### 💡 안드로이드 SAMF 구조와의 비교 (정확한 비유)
* 안드로이드의 **Single Activity Multiple Fragments (SAMF)** 구조와 완벽히 유사합니다.
* 전체 생명주기를 관장하고 하우징 역할을 하는 큰 그릇인 `Activity`(하나의 정적 `index.html`)가 화면에 떠 있고, 그 내부 영역을 사용자의 이동 흐름에 따라 `Fragment`(자바스크립트가 주입하는 React Component) 단위로 떼었다 붙였다(Mount/Unmount) 하는 구조와 일맥상통합니다.
* **라우팅(Routing)**: 웹 브라우저의 기본 페이지 새로고침 동작 없이, 자바스크립트 내장 History API(또는 React Router 등)를 사용해 브라우저의 주소창 URL만 조작하고 내부 컴포넌트 트리를 동적으로 치환해 줍니다.

### ⚖️ 장단점
* **장점**: 초기 로딩 이후에는 화면 전환 시 깜빡임(화면 전체 새로고침)이 없고, 필요한 API JSON 데이터만 서버에서 조회하므로 네이티브 앱과 같은 매끄럽고 빠른 사용자 경험(UX)과 인터랙션을 제공합니다.
* **단점**: 첫 방문 시 어플리케이션 구동에 필요한 무거운 자바스크립트 번들 파일 전체를 받아와야 하므로 **초기 구동 속도가 느립니다 (Slow FCP)**. 또한 초기 서버 응답 HTML이 텅 빈 구조(`<body><div id="root"></div></body>`)이기 때문에 검색엔진 봇이 정보를 긁어가기 어려워 **검색엔진 최적화(SEO)에 불리**합니다.

## 2. SSR (Server Side Rendering)

SSR은 브라우저가 페이지를 요청할 때마다 **서버에서 실시간으로 데이터베이스를 조회하고 관련 HTML을 완전히 렌더링(완성형 HTML)하여 브라우저에 즉시 전달**하는 전통적이고도 진화된 방식입니다.

### ⚠️ 오해 바로잡기: SSR의 "보이기만 하고 먹통인 시기" 원인
* **잘못 알고 있었던 점**: "초기 화면은 빠르나 버튼 입력이 작동 안 하는 시기가 있는 이유는 버튼 클릭 시 서버와 통신해서 갔다 와야 하기 때문이다."
* **실제 컴퓨터 공학적 이유**:
  * 서버가 완성된 HTML을 즉시 보냈기 때문에 사용자는 뼈대와 글자가 그려진 화면을 빠르게 봅니다. 이 시점을 **TTV (Time To View, 화면이 보이는 시점)**라고 합니다.
  * 하지만 이 시점에는 화면에 버튼 태그는 그려져 있지만, 그 버튼에 클릭 이벤트 리스너(자바스크립트 코드)를 바인딩해 줄 **자바스크립트 번들이 아직 브라우저에 다운로드되거나 실행 완료되지 않은 상태**입니다.
  * 자바스크립트 번들이 다운로드 완료되어 기존 정적 HTML 뼈대 위에 리액트의 인터랙티브 로직을 주입하고 핏줄을 잇는 과정을 **수화(Hydration)**라고 합니다.
  * 즉, **TTV(화면 보임)와 TTI(Time To Interactive, 상호작용 가능 시점) 사이에 존재하는 시간적 공백기**는 서버와의 실시간 네트워크 통신 비용 때문이 아니라, **Hydration(자바스크립트 바인딩) 연산이 아직 끝나지 않았기 때문**에 발생합니다.

```
[클라이언트 요청] ➔ [서버가 HTML 반환] ➔ [TTV: 화면 표시 (인터랙션 불가)] ➔ [JS 번들 로드] ➔ [Hydration 완료] ➔ [TTI: 상호작용 가능]
```

### ⚖️ 장단점
* **장점**: 완성된 HTML을 반환하므로 크롤러 봇이 내용을 완벽히 수집하여 **SEO에 매우 유리**하며, 빈 화면 노출 시간이 짧아 사용자 체감 속도(TTV)가 매우 빠릅니다.
* **단점**: 매 요청마다 서버에서 렌더링 연산을 수행하므로 서버의 CPU 부하가 증가하고, TTV와 TTI 사이의 입력 먹통 구간이 존재할 경우 사용자가 버튼을 눌러도 반응하지 않는 불쾌한 사용자 경험을 겪을 수 있습니다.

## 3. SSG (Static Site Generation)

SSG는 동적으로 요청 시점에 렌더링을 하는 것이 아니라, **어플리케이션을 빌드하는 단계(Build Time)에서 필요한 데이터 조회를 마치고 정적 HTML 파일들을 미리 모조리 만들어두는 기법**입니다.

### ⚖️ 장단점
* **장점**: 사용자가 요청하면 미리 서버(혹은 CDN)에 물리 파일로 존재하던 완성형 HTML을 그대로 바로 내보내 주기 때문에 **세 가지 패턴 중 속도가 가장 압도적으로 빠르며**, 서버 연산 부하가 거의 없음에도 전 세계 어디서나 최적의 속도로 배포(CDN 캐싱)하기 편리합니다. 당연히 완성형 HTML이므로 **SEO에도 매우 탁월**합니다.
* **단점**: 한 번 빌드하면 내용이 거의 변경되지 않는 정적 문서(블로그, 공식 문서, 제품 소개 페이지 등)에만 제한적으로 활용할 수 있습니다. 마이너한 텍스트 한 줄을 바꾸기 위해서도 저장소 전체 프로젝트의 재빌드 및 재배포(Re-build) 과정을 거쳐야 하는 관리적 한계가 존재합니다.

## 4. SPA, SSR, SSG 최종 비교표

| 지표 / 패턴 | SPA (Client-Side Rendering) | SSR (Server-Side Rendering) | SSG (Static Site Generation) |
| :--- | :--- | :--- | :--- |
| **HTML 생성 시점** | 브라우저 런타임 (JS 실행 시) | 매 사용자 요청 시점 (On-demand) | 빌드 시점 (Build Time) |
| **초기 로딩 속도 (TTV)** | 🔴 느림 (JS 로딩 지연) | 🟢 매우 빠름 | 🥇 가장 압도적으로 빠름 |
| **초기 인터랙션 (TTI)** | 🟢 로드 즉시 가능 (TTV = TTI) | 🟡 지연 존재 (TTV < TTI) | 🟡 지연 존재 (TTV < TTI) |
| **페이지 전환 속도** | 🥇 매우 빠르고 부드러움 | 🔴 새로고침 발생 (서버 왕복) | 🔴 새로고침 발생 (서버 왕복) |
| **서버 부하** | 🟢 거의 없음 (클라이언트가 연산) | 🔴 높음 (매 요청 연산) | 🟢 거의 없음 (정적 파일 응답) |
| **SEO 최적화** | 🔴 나쁨 (자바스크립트 의존) | 🟢 매우 좋음 (완성형 HTML) | 🟢 매우 좋음 (완성형 HTML) |
| **적합한 서비스** | 대시보드, 어플리케이션, 네이티브 앱 UX | 커뮤니티, 뉴스, 실시간 상품 조회몰 | 블로그, 포트폴리오, 공식 소개 사이트 |

## 5. React Server Components (RSC) vs Client Components (RCC)

가장 최신의 React 아키텍처는 서버 렌더링의 이점과 클라이언트 인터랙션의 장점을 하이브리드하여 극대화하기 위해 컴포넌트를 실행 영역 기준으로 2가지로 나눕니다.

### 5.1 React 서버 컴포넌트 (RSC, React Server Components)
* **핵심 정의**: **서버 환경에서만 실행되고 물리적으로 해석되는 컴포넌트**입니다. 
* **동작 원리**: 
  * 서버 컴포넌트의 자바스크립트 소스 코드는 **브라우저에 아예 전달되지도 않으며 브라우저 엔진이 이를 실행하지도 않습니다.** 
  * 서버 컴포넌트가 서버에서 실행되어 반환하는 최종 결과물은 원시 HTML이 아닌, 가상 DOM 노드를 직렬화하여 표현한 **RSC Payload (JSON 형태의 트리 데이터 구조)**입니다. 브라우저는 이 Payload를 읽어 가상 DOM에 병합하고 렌더링에 활용합니다.
* **장점**: 브라우저로 불필요한 라이브러리나 자바스크립트 코드가 날아가지 않아 **프론트엔드 번들 사이즈를 극적으로 줄일 수 있습니다.** 
* **제약**: 서버에서만 돌기 때문에 버튼 클릭 이벤트 바인딩(`onClick` 등)이나 클라이언트 상태/이펙트 Hook(`useState`, `useEffect` 등)을 사용할 수 없습니다.

### 5.2 React 클라이언트 컴포넌트 (RCC, React Client Components)
* **핵심 정의**: 기존 React 앱처럼 브라우저에서 동작하며 **사용자 인터랙션(이벤트, 상태 관리)을 전담하는 컴포넌트**입니다. 파일 최상단에 `"use client"` 지시어를 작성하여 활성화합니다.
* **동작 메커니즘과 이중 실행(Double Execution) 비효율**:
  * Next.js 등 SSR 프레임워크 환경에서 클라이언트 컴포넌트 또한 **최초 로딩 시에는 서버에서 HTML로 컴파일(SSR)되어 브라우저에 빠르게 전달**됩니다. (즉, 클라이언트 컴포넌트도 서버에서 실행됩니다)
  * 이후 브라우저가 이를 다운받아 화면에 그린(TTV) 뒤, 리액트가 정적 HTML과 가상 DOM을 매칭하며 이벤트 핸들러를 잇는 **수화(Hydration) 단계에서 클라이언트 컴포넌트를 브라우저 단에서 똑같이 한번 더 실행**합니다.
  * 결과적으로 **서버와 브라우저 양쪽 환경에서 컴포넌트 코드가 모두 실행되는 이중 연산(Double Execution)의 비효율**이 발생하며, 브라우저가 관련 컴포넌트 코드를 전부 다운받아야 하므로 번들 사이즈가 커지는 오버헤드를 동반합니다.

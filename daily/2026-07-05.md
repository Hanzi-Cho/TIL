# TIL [2026-07-05] - STAR 기법 적용

## Situation & Task (상황 및 문제 정의)
* **상황**: 매일 작성하는 TIL(Today I Learned) 문서를 Git 저장소에 커밋하고 푸시하는 과정이 번거롭고 누락될 위험이 있음.
* **목표**: 매일 특정 시간에 스스로 깃 커밋 및 푸시가 가동되도록 크론(Cron) 자동화를 시도하고, 최종적으로 가장 확실하고 직관적인 수동 실행형 래퍼 스크립트를 구축하는 것.

## Action (원인 분석 및 해결 과정)
* **1단계: 크론탭(Crontab) 스케줄러 등록**
  * 매일 밤 11시 50분(`50 23 * * *`)에 동작하는 자동 커밋/푸시 쉘 스크립트(`auto_push.sh`)를 작성하여 크론에 등록함.
* **2단계: 컴퓨터 전원 상태에 따른 크론의 한계 인식 및 피드백**
  * 개인 PC나 노트북이 꺼져 있거나 절전 모드(Sleep) 상태일 때 크론 스케줄이 무시된다는 사실을 검토.
  * `@reboot` 방식이나 `anacron` 대안이 있으나, 사용자의 요구에 맞게 크론 설정을 해제하고 `til` 프로젝트 루트 경로에서 `./push` 명령어 하나로 실행되는 래퍼 스크립트 구조로 전환하기로 결정.
* **3단계: 원격 커밋 충돌 방지 및 안전성 개선**
  * GitHub 원격 저장소에 주기적인 자동화 커밋(예: `skip ci`)이 비동기적으로 발생하여 로컬 push가 거절당하는 현상 발견.
  * `auto_push.sh` 내부에서 push를 진행하기 직전에 자동으로 원격 변경사항을 내려받아 합치도록 `git pull --no-rebase --no-edit origin main` 로직을 추가하여 안전한 선제 병합 처리.
  * `auto_push.log` 파일이 Git 추적 대상에 들어가지 않도록 `.gitignore` 파일을 작성하여 예외 처리.

## Result (결과 및 정량적 지표)
* 크론탭 스케줄러 의존성을 걷어내고, 사용자가 원하는 시점에 언제든 `./push` 명령어 단 한 번만으로 로컬 변경 사항 커밋, 원격 비동기 이력 병합, 원격 저장소 푸시까지의 3단계를 안전하게 자동 수행 완료.
* 불필요하게 생성되던 `auto_push.log` 파일도 Git 추적에서 깔끔히 격리됨.

## Takeaway (인사이트)
* 리눅스 `cron` 데몬은 예약 시간 당시 시스템이 활성화되어 있지 않으면 소급 적용하지 않고 스킵하므로, 항상 켜져 있는 서버가 아닌 개인 데스크톱/노트북 환경에서는 `@reboot`나 온디맨드 스크립트 실행이 더 직관적인 해결책이 될 수 있음.
* 협업 및 봇 자동화가 연동된 저장소 환경에서는 로컬 커밋을 강제로 올리기 전에 항상 원격 브랜치를 선제적으로 `pull` 하여 병합하는 방어적 스크립트 설계가 필수적임.

# TIL [2026-07-05] - Janggi 프로젝트 Expo 구동 + 테스트 아키텍처 + 변형 게임 초석

## Situation & Task (상황 및 문제 정의)
* `workspace/game/janggi` 프로젝트를 검토한 결과, `CLAUDE.md`/`docs/modification-pipeline.md`가 "docs·design-system 수정 → 코드 반영 → 검증"이라는 7단계 절차를 문서로는 촘촘하게 정의해두었지만, 이를 강제하는 자동화 장치(테스트, lint, CI)가 전혀 없었음.
* 실제로 `design-system/tokens.json`은 어떤 코드에서도 import되지 않았고, `App.tsx`(단일 659줄)는 GUARDRAIL(GR-01/02/05/07 등)을 광범위하게 위반한 채 하드코딩 색상/폰트와 UI 레이어의 게임 로직 재구현으로 채워져 있었음.
* 목표: Expo로 실제 구동 가능한 상태를 만들고, 검증 가능한 테스트 아키텍처를 갖추고, "변형 장기(미니 장기, 포 제한 해제 등)"를 실제로 추가하기 쉬운 구조적 기반을 다지는 것.

## Action (원인 분석 및 해결 과정)
* `docs/modification-pipeline.md` STEP5 체크리스트에 빠져있던 GR-01~06(디자인 토큰 규칙) 항목을 추가해, "디자인 시스템 변경 검증"이 실제로 체크리스트에 포함되도록 보강.
* `tokens.json`을 실제로 소비하는 `src/ui/design-system/theme.ts`를 신설하고 `App.tsx`의 하드코딩 색상/폰트를 전부 교체하는 과정에서, 부수적으로 두 개의 숨은 버그를 발견해 함께 수정함.
  * `tsconfig.json`의 `include`가 `src/**/*`만 지정되어 있어 `tsc --noEmit`(STEP6 빌드 검증)이 실제 엔트리 파일인 `App.tsx`를 아예 검사하지 않고 있었음.
  * 위 설정을 고치자마자 RN에 존재하지 않는 브라우저 전역 `alert()` 호출 버그가 즉시 드러남 → `Alert.alert()`로 교체.
* `App.tsx` 단일 파일을 `src/ui/screens/`(SetupScreen, PlayScreen)와 `src/ui/components/`(BoardGrid, PieceView, LayoutPicker, BikjangOverlay, CapturedList)로 분해. UI가 직접 `Board`/`evaluator`를 조작해 자살수 시뮬레이션을 재구현하던 GR-07 위반은 `GameSession.getLegalMoves()`라는 단일 API로 승격.
* `GameSession` 생성자에 `boardConfig?: BoardConfig` 주입 포인트를 추가 — 이전에는 `new Board()`가 항상 표준(9x10) 설정으로 고정되어 있어 "변형 게임 = 신규 파일 1개, 기존 코드 무수정"이라는 아키텍처 문서의 주장이 실제로는 성립하지 않았음을 확인하고 해소.
* `StandardEvaluator`(기물 7종 이동 로직이 한 클래스에 몰려 356줄) 를 `IPieceMovement` 전략 패턴(`src/rules/standard/pieces/*`)으로 분리(ADR-003). 생성자에서 기물 하나만 오버라이드하면 나머지 6개는 표준 규칙을 그대로 재사용할 수 있게 되어, "포 제한 해제 장기" 같은 변형을 실제로 저비용으로 만들 수 있는 구조로 전환.
* `jest` + `jest-expo`를 처음 도입하고 `Board`/`StandardEvaluator`/`GameSession`에 단위 테스트 47개를 작성. 이전까지 테스트가 0개였고, `docs/pipeline-execution-log.md`의 STEP5 "PASS" 기록은 전부 사람이 눈으로 보고 적은 자기서술(self-report)이었음을 확인.
* `CLAUDE.md`/`modification-pipeline.md`의 빌드 검증(STEP4/6) 항목에 `npm test` 실행을 추가하고, `CHANGELOG.md`/ADR-003 문서화, `docs/rules.md`(참조되지 않던 구버전 규칙 문서) 정리까지 마무리.

## Result (결과 및 정량적 지표)
* `npx tsc --noEmit` 통과, `jest` 단위 테스트 47/47 통과, `npx expo export -p web`으로 실제 웹 번들(약 364KB, 165 모듈) 생성까지 확인 — "빌드는 되지만 실제로 안 켜본" 상태에서 실제 구동 가능한 상태로 전환.
* 변형 게임 추가 시 실제로 건드려야 할 파일 범위(보드 크기 주입, 기물 하나만 오버라이드)를 코드로 검증 가능한 수준까지 좁힘.

## Takeaway (인사이트)
* 문서 기반 파이프라인(`CLAUDE.md`, 체크리스트)은 자동 검증 장치가 없으면 "PASS" 기록이 결국 자기서술에 그친다 — 실제 `tsc`/`jest` 실행 결과가 있어야 신뢰할 수 있다.
* `tsconfig.json`의 `include` 설정 하나가 엔트리 파일을 빠뜨리면, 그 위에 아무리 정교한 "빌드 검증 STEP"을 문서로 적어놔도 실질적으로 아무것도 검증되지 않는다.
* 전략 패턴(`IPieceMovement`)으로 기물별 규칙을 분리하면 변형 게임에서 "기물 하나만 바꾸고 나머지는 재사용"이 가능해져 실질적인 개방-폐쇄 원칙(OCP)을 달성할 수 있다.
* 아키텍처 문서의 "확장성 주장"(예: 변형 게임 추가 시 파일 1개만 필요)은 실제 생성자 시그니처·초기화 로직을 직접 열어보기 전까지는 검증된 사실이 아니다 — 문서와 코드 사이의 괴리는 항상 의심하고 직접 확인해야 한다.

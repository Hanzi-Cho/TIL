# TIL [2026-07-08] - STAR 기법 적용

## Situation & Task (상황 및 문제 정의)
* **상황**: `blaster-launcher` 프로젝트의 웹 프론트엔드 환경에서 잘 작동하던 `REPORT DOWNLOAD PDF` (PDF 보고서 다운로드) 기능이 Android WebView 런처 앱 환경에서 다운로드가 유실/실패하는 크리티컬 버그 감지.
* **원인 분석 대상**:
  * PDF 파일을 서빙하는 서버 API가 인증(`Bearer <Token>`)을 요구함에 따라, 단순한 HTML `<a>` 태그 링크로는 다운로드가 불가. (요청 헤더 주입 불가능 제약)
  * 이를 우회하고자 프론트엔드 단에서 `axios`를 통해 헤더를 실어 요청하고 응답 데이터(바이너리)를 브라우저 메모리에 `Blob` 객체 형식으로 복원하여 받고 있었음.
* **해결 태스크**:
  * Web 브라우저 내부 메모리에 갇혀 있는 가상의 임시 URL(`blob:`) 구조를 Android OS 시스템 레벨의 다운로더가 어떻게 감지하고 예외를 내는지 조사.
  * WebView 단방향 샌드박스를 넘어 Android 네이티브 파일 시스템에 안전하게 기록할 수 있는 하이브리드 연동 규격 설계 및 도입.
* **[추가 태스크]**: 면접 대비 컴퓨터 공학 기초 지식 점검을 위해 OS 동시성 개념(프로세스 vs 스레드 차이, 데드락 발생 4대 조건) 및 네트워크 프로토콜(OSI 7계층, TCP vs UDP 특성 및 실시간 제어에서의 선택 기준, HTTP vs HTTPS 차이 및 TLS 핸드쉐이크 절차) 학습 및 정리 진행.
* **[추가 태스크 2]**: 선형 자료구조(배열 vs 연결 리스트, 스택 vs 큐의 물리적 특성 및 최적 구현체 선택 기준) 및 5대 정렬 알고리즘(버블, 선택, 삽입, 퀵, 병합 정렬 복잡도 및 기작) 학습 및 정리 진행.

## Action (원인 분석 및 해결 과정)
* **임시 Blob URL의 물리적 실체 파악**:
  * 브라우저에서 `URL.createObjectURL(blob)`로 얻어지는 `blob:http://...` 주소는 실제 서버 주소가 아닌, **브라우저 탭의 V8 Javascript 힙 메모리** 내부에 바인딩된 이진 객체 포인터(Reference)임을 인지.
* **Android WebView Downloader의 동작 한계 규명**:
  * Android WebView 내부에서 다운로드 이벤트가 감지되어 Android OS의 `DownloadManager`에 처리를 위임할 때, `DownloadManager`는 독자적인 OS 네트워크 스택을 사용해 해당 URL로 새 HTTP 요청을 시도함.
  * 하지만 `blob:~~` 주소는 브라우저 내부 메모리에만 존재하는 가상 영역이므로 OS 레벨의 네트워크 호출로 해석할 수 없어 커널 단에서 거절(Rejection) 처리됨을 분석함.
* **Base64 postMessage 브릿지 구조 설계 및 구현**:
  * **웹(Web-side) 처리**:
    * Axios를 통해 가져온 Binary `Blob` 데이터를 `FileReader.readAsDataURL()`을 이용하여 문자열 형태의 `Base64` 포맷으로 스트리밍 변환.
    * 변환된 Base64 페이로드 및 메타데이터(파일명, MIME 타입)를 JSON으로 묶어 `window.ReactNativeWebView.postMessage()` 브릿지를 통해 React Native 영역으로 전송.
  * **네이티브(React Native-side) 처리**:
    * WebView의 `onMessage` 속성으로 수신 이벤트를 파싱.
    * `react-native-fs` 라이브러리를 활용해 Base64 문자열을 원천 바이너리로 복원하여 런처의 로컬 파일 시스템(`RNFS.DocumentDirectoryPath`)에 직접 기입.
    * 파일 쓰기 완료 후 `react-native-file-viewer`를 가미하여 OS 뷰어 호출 및 즉각 오픈 연동을 지원함으로써 UX를 크게 극대화함.
* **OS 동시성 핵심 이론 복습**:
  * 프로세스의 메모리 격리(가상 주소 영역 독립)와 스레드의 자원 공유(Code, Data, Heap 영역 공유 및 Stack/PC 개별 할당)의 물리적 메모리 지도를 비교 정립함.
  * 프로세스 Context Switch 시 MMU 페이지 테이블 교체 및 TLB 캐시 전체 플러시가 발생하는 오버헤드를 분석하여 멀티스레드의 가벼움을 논증함.
  * 데드락(Deadlock)의 4대 필요조건인 상호 배제(Mutual Exclusion), 점유 대기(Hold and Wait), 비선점(No Preemption), 순환 대기(Circular Wait) 개념을 정리함.
* **네트워크 프로토콜 아키텍처 정리**:
  * OSI 7계층의 헤더 캡슐화/역캡슐화 방향성 및 전송 흐름을 도식화하고, TCP와 UDP의 전송 신뢰성/오버헤드 특징을 매핑함.
  * 특히 패킷 유실 시 재전송 대기로 지연(HOL Blocking)이 생기는 TCP 대신 최신 패킷을 즉시 주입할 수 있는 UDP가 실시간 로보틱스 제어 및 실시간 게임 통신에 왜 적합한지 원리를 규명함.
  * HTTP의 평문 도청 취약성을 보완하기 위한 HTTPS(TLS) 보안 기작을 학습함. TLS 핸드쉐이크 과정을 6단계로 정리하고, 비대칭키를 사용하여 대칭키를 안전하게 교환하는 하이브리드 설계의 가치를 기술함.
* **선형 자료구조 특성 비교**:
  * 물리적 메모리가 연속적인 Array(O(1) 임의 접근, O(n) 중간 삽입/삭제)와 비연속적인 LinkedList(O(n) 탐색, O(1) 삽입/삭제)의 성능 균형을 대조함.
  * 삽입/삭제가 맨 뒤에서만 일어나는 Stack은 Array 구현이 최적이고, 맨 앞 제거가 핵심인 Queue는 LinkedList 구현이 필수적임을 논증함.
* **정렬 알고리즘 이론 정립**:
  * 5대 정렬의 평균/최악 복잡도를 비교하고, 정렬 전후 데이터 순서가 보존되는 Stable 여부와 가상 임시 공간(O(n))을 요구하는 Merge Sort의 물리적 한계를 이해함. 퀵 정렬의 최악 조건($O(n^2)$) 및 방지법을 정리함.

## Result (결과 및 정량적 지표)
* **결과**: Android WebView 런처 환경에서 인증 토큰이 적용된 PDF 보고서 파일이 드롭 없이 100% 정상 다운로드 및 실시간 뷰어 보기 연동이 완료됨을 확인함.
* **지식 문서 신설**: 웹뷰 환경에서의 Blob 다운로드 기법을 정리한 [webview-blob-download.md](../knowledge/react-native/webview-blob-download.md) 지식 문서를 신설 확보하여 향후 하이브리드 연결 패턴의 표준 스펙으로 설정함.
* **동시성, 네트워크 및 알고리즘 지식 확장**: 학습한 OS 핵심 가치, 네트워크 표준 개념 및 핵심 자료구조/정렬 알고리즘을 각각 [process-thread-deadlock.md](../knowledge/concurrency/process-thread-deadlock.md), [network-protocols.md](../knowledge/network/network-protocols.md) 및 [data-structures-sorting.md](../knowledge/algorithm/data-structures-sorting.md) 지식 문서로 정형화하여 면접 대비 포트폴리오 기틀을 다짐.

## Takeaway (인사이트)
* **메모리 포인터와 샌드박스의 한계**: `blob:` URL 스키마는 웹뷰 컨테이너 내부의 격리된 메모리 세션에서만 유효한 논리 주소이므로, OS 커널 수준의 네트워크 스택을 타는 `DownloadManager`로는 직접 접근할 수 없다는 기본 원리(First Principles)를 학습함.
* **하이브리드 브릿지의 필요성**: 웹 표준 API와 네이티브 OS 영역의 권한 및 파일 저장 구조를 중재하기 위해, 직렬화된 데이터(Base64) 스트림을 브릿지로 우회 전달하여 기기 물리 저장소에 마운트시키는 연동 최적화의 가치를 이해함.
* **컴퓨터 공학 물리 계층의 연계성**: 실시간 통신 및 동시성 제어 시 단순히 논리적 코딩만 수행하는 것을 넘어, 하드웨어 레벨의 Context Switch 비용(TLB 플러시), 전송 계층 지연(TCP HOL Blocking), 비대칭 암호 연산 오버헤드 등 기본 물리 법칙과 커널 동작 관점을 함께 고려해야 병목 없는 완성형 하이브리드 설계가 가능해짐을 깨달음.
* **자료구조의 물리 구조 지향**: 추상적인 자료구조 선택이 물리적인 메모리 배치 구조(연속성 유무) 및 CPU 캐시 지역성에 의존하여 런타임 성능 편차를 결정하므로, 하드웨어 친화적(Hardware-friendly) 설계를 지향해야 함을 배움.

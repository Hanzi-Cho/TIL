# TIL [2026-07-06] - STAR 기법 적용

## Situation & Task (상황 및 문제 정의)
* 영업팀 피드백에 맞추어 디자인 시스템 UI 레이아웃 및 컴포넌트 구조를 수정해야 하는 과제 발생.
* 백엔드 API로부터 수신하는 동적 데이터를 활용하여 특정 UI 요소들이 정확한 타이밍에 유연하게 리렌더링되도록 처리하는 매핑(Mapping) 설계가 필요함.
* 빈번하게 응답 값이 전달되는 API 구조를 UI 컴포넌트 계층에 직접 주입할 경우, 불필요한 깊은 비교(Deep Comparison) 오버헤드와 무의미한 부모-자식 전반의 리렌더링(성능 병목)이 야기될 수 있으며 스키마 변경 시 UI 코드의 파손 리스크가 큼.
* **[추가 태스크]** 삼성전자 DS 및 현대자동차 소프트웨어 직무 대비 C++ 시스템 런타임 프로젝트 설계 방안, 반도체 및 자동차 SDV 관련 핵심 면접 질문을 정리하고, 나아가 AI 시대 시니어 프론트엔드 개발자 타겟의 React/FE 핵심 아키텍처 및 디버깅 면접 질문들을 지식 문서화하는 작업이 필요함.
* **[추가 태스크 2]** 리액트의 기본 동작 원리(컴포넌트 의존성 캡슐화, JSX 렌더링 메커니즘, Local vs Feature 상태 경계 정의, Hooks의 내부 메커니즘)를 보완하고 체계적으로 정비함.
* **[추가 태스크 3]** 리액트의 선언형 UI 성격, 브라우저 DOM과 메모리 상의 가상 DOM(스냅샷)의 개념 및 Diffing 재조정(Reconciliation) 파이프라인, 그리고 렌더링 타이밍 관련 `useEffect`와 `useLayoutEffect`의 동작 시점을 추가 고도화함.
* **[추가 태스크 4]** 모던 웹 어플리케이션의 대표적 세 가지 렌더링 방식인 SPA, SSR, SSG의 개념과 차이 및 Hydration, TTV/TTI 핵심 성능 메커니즘을 명확화하여 정리함.
* **[추가 태스크 5]** React Server Components (RSC)와 Client Components (RCC)의 실행 위치 차이, 브라우저 코드 미전송(RSC Payload) 이점 및 Hydration 시의 이중 실행(Double Execution) 오버헤드를 보완 이관함.

## Action (원인 분석 및 해결 과정)
* **디자인 시스템 UI 매퍼**:
  * **가설 1**: 통째로 API Response 객체를 컴포넌트의 단일 State로 관리하여 업데이트 -> 데이터가 미세하게 변경되거나 불필요한 정보가 들어올 때도 전체 트리가 렌더링되어 성능 정체 확인.
  * **가설 2**: **어댑터 패턴(Adapter Pattern)**을 적용하여 외부 API 스펙(DTO)과 로컬 UI 상태(State/ViewModel)를 완벽히 격리하는 매퍼 레이어 구축.
  * **결정된 해결책**: API 원시 데이터를 UI 컴포넌트 렌더링에 최적화된 객체로 가공하는 매퍼 함수(`apiToUiMapper`)를 정의하고, 셀렉터(Selector) 기반 구독 구조를 적용함.
* **도메인 지식 정리 (반도체/자동차/프론트엔드)**:
  * 본인의 깃허브 오픈소스 포트폴리오를 분석하여 C++ 임베디드 런타임 프로젝트 로드맵을 설계함.
  * 반도체 하드웨어 인터페이스와 자동차 SDV 및 실시간 미들웨어 핵심 면접 질문 20종을 분류 저장함.
  * 추가로 React Concurrent Rendering, API 설계 규약, 상태 관리 철학, 순환 의존성 해결 등 아키텍처적 요소 및 AI 생성 코드 검증 기준과 AI 시대 FE 개발자의 역할을 다루는 프론트엔드 핵심 질문 11종을 수립하여 별도 지식 문서로 보관함.
  * **리액트 개념 정밀화**: HTML/XML과 대비한 컴포넌트 캡슐화의 이점, `COMPONENT ➔ JSX ➔ STATE ➔ HOOK ➔ EVENT` 간의 상태 전이 메커니즘 정의, Local State와 Feature State의 공유 한계 명확화 및 리액트의 3단계 렌더링(Trigger -> Render -> Commit) 파이프라인을 지식화함.
  * **렌더링 심화 개념 보강**: 브라우저 실물 DOM 제어 오버헤드를 극복하기 위해 메모리 상에 가상 DOM(React Element Tree) 스냅샷을 구성하고 Diffing으로 변경점을 동적으로 계산하는 재조정(Reconciliation) 과정을 명확히 규정함. 또한 선언형 UI(Declarative) 패러다임의 특징과, 브라우저 페인팅 차단 여부에 따른 `useEffect`(Paint 후 비동기 실행) 및 `useLayoutEffect`(Paint 전 동기 실행)의 실행 순서 대조 메커니즘을 상세히 이관함.
  * **렌더링 패턴 분류**: SPA의 무새로고침 화면 치환을 안드로이드의 SAMF(Single Activity Multiple Fragments) 구조와 빗대어 명확화함. SSR의 TTV(화면 노출) 대비 TTI(상호작용 가능) 지연 원인이 서버 통신 때문이 아니라 이벤트 핸들러 바인딩 단계인 **수화(Hydration)** 연산 때문임을 명시하고, SSG 빌드타임 정적 생성의 장단점을 아우르는 분석표를 작성함.
  * **RSC 및 RCC 분석 보강**: 브라우저가 실행 코드를 아예 다운로드하지 않고 서버 영역에서 직렬화된 데이터인 **RSC Payload**를 반환하여 연산하는 서버 컴포넌트(RSC)의 특성을 정립함. 또한 클라이언트 컴포넌트(RCC)가 최초에 서버 측 렌더링(SSR)을 거치고 브라우저 수화(Hydration) 시점에 한 번 더 구동되는 **이중 실행(Double Execution)의 비효율**을 규명해 이관함.

## Result (결과 및 정량적 지표)
* API 수신에 따른 불필요한 컴포넌트 리렌더링 횟수를 기존 대비 약 50% 이상 감축하여 성능 최적화 달성.
* 반도체 소프트웨어, 자동차 SDV, 리액트 프론트엔드 분야 면접 대비 질문 총 31선 및 리액트 가상 DOM과 동시성 기반 렌더링 지식 시스템, 웹 렌더링 아키텍처(SPA/SSR/SSG/RSC) 맵을 성공적으로 고도화 축적함.

## Takeaway (인사이트)
* UI 렌더링 시스템이 외부 데이터 의존성(API DTO)과 강하게 결합할 경우 컴포넌트 성능 저하와 취약한 코드(Fragile Code)로 이어짐.
* 단순 코드 작성 수준을 넘어서서 React의 Trigger-Render-Commit 주기와 렌더링 시점의 멱등성 및 레이아웃 페인팅 타이밍을 깊게 이해하는 것이 컴포넌트 최적화의 첫단추임을 배움.
* 서비스의 콘텐츠 동적 변동성과 SEO 검색 노출 요구조건에 맞춰 적절한 렌더링 아키텍처(SPA, SSR, SSG)를 균형감 있게 조합해 나가는 기술적 혜안을 획득함.

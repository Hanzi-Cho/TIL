## Part 1. Design System & Google Stitch MCP 연동

### Situation & Task (상황 및 문제 정의)
* Claude Design을 통해 상태 관리가 적용된 인터랙티브 UI 생성을 시도했다.
* 이어서 AI 에이전트(Claude Code 등)가 직접 Google Stitch를 조작하여 디자인 작업을 자동 수행할 수 있도록 MCP(Model Context Protocol) 연동을 추진했다.
* 연동 도중 local 기반 MCP 설정과 http(네트워크) 기반 MCP 설정의 구성 방식 차이로 인해 연결 에러가 발생하는 문제를 겪었다.
* 모니터링 시스템의 화면별 전반적인 요구사항을 상세히 정의하고 관리하기 위해, 프로젝트 내 `docs/spec` 경로에 각 스크린별 명세(구성요소, 규칙, 하지 말아야 할 것, 레이아웃 원칙 등)를 표준 문서화해야 했다.

### Action (원인 분석 및 해결 과정)
* local MCP와 http MCP의 환경 설정 포맷이 다름을 확인하고 두 구성 방식을 비교 분석했다.
* 로컬 실행 방식이 아닌, 네트워크 상의 HTTP MCP 사양에 맞게 서버 설정 및 포트 환경 변수를 다시 매핑했다.
* 구글 스티치를 네트워크 기반 MCP로 정상 연결하도록 주소를 바로잡아 통신 오류를 해결했다.
* 기존 `monitoring-screen-spec-claude-design.md` 문서를 정밀 분석하여, 공통 레이아웃 원칙과 5개의 세부 상태/패널 스펙으로 구조를 분할 및 정규화하여 신규 문서들을 작성했다.

### Result (결과 및 정량적 지표)
* Claude Code - Google Stitch 간의 MCP 연동을 무사히 완료했다.
* AI 에이전트가 직접 도구 도메인 API를 호출하여 Google Stitch 내 UI 요소를 조작할 수 있는 자동화 파이프라인을 구축했다.
* `docs/spec/` 디렉토리 하위에 공통 원칙(README.md) 및 5개 핵심 세부 사양서(통신 설정, 무장, 완료, 취소, 상세 패널) 등 총 6개의 마크다운 명세 문서를 성공적으로 구축했다.

## Part 2. 안드로이드 11 전용 RN 프로젝트 및 Webview 프론트 Android 8.1 호환 적용

### Situation & Task (상황 및 문제 정의)
* **목표**: 안드로이드 11(API 30) 타겟으로 작성된 기존 React Native 프로젝트 및 `blaster-web` 프론트엔드 소스코드를 안드로이드 8.1(API 27, TC56 장비) 환경에 호환 가능하도록 이식 및 배포해야 했다.
* **문제 현상**: 
  * 구형 디바이스인 TC56에서 빌드된 하이브리드 웹뷰를 구동하면 화면이 정상적으로 렌더링되지 않고 **화이트스크린(White Screen)**만 노출되는 에러가 발생했다.
  * 구형 기기 배포를 위해 네이티브 수준의 `minSdkVersion` 하향과 그에 따른 런타임 권한 획득 처리, 그리고 구형 크롬 웹뷰 렌더링 호환성 점검이 동시에 요구되었다.

### Action (원인 분석 및 해결 과정)
* **수동 빌드 및 파일 교체 배포**:
  * 로컬 환경에서 `blaster-web`을 빌드한 후 `build` 디렉토리를 `public.tar.gz`로 압축하여 scp를 통해 대상 장비의 `/tmp` 디렉토리에 업로드했다.
  * ssh로 장비에 접속한 후 root 권한(`sudo -s`)을 획득하여 `/HOST/bin/blaster-system/public` 경로의 기존 정적 자원을 백업하고, 업로드한 새 빌드 본으로 교체했다. (해당 장비는 Docker/Nginx 프록시가 아닌 정적 파일 직접 연동 아키텍처임 확인)
* **배포 확인 및 파일 정합성 확인**:
  * 장비 측의 `asset-manifest.json`, `index.html`, `static/js/main.*.js` 파일 목록을 확인하고 로컬에서 측정한 sha256 해시값과 대조하여 프론트 파일 자체는 문제없이 배포되었음을 확인했다. 즉, 배포 실패가 아닌 **웹 엔진 실행 중 런타임 크래시** 상태임을 규명했다.
* **Android 네이티브 minSdk 조정 및 권한 분기**:
  * React Native 빌드 환경에서 구형 OS를 호환하도록 `minSdkVersion`을 하향 패치했다.
  * OS API 버전에 맞춰 빌드 플레이버(Build Flavor)별 권한 및 런타임 권한 분기 로직(Build.VERSION.SDK_INT 조건 분기)을 적용하여 구형 환경에서도 앱이 비정상 종료되지 않도록 예외 처리했다.
* **Chrome 호환성 및 globalThis 에러 진단**:
  * `package.json`의 `browserslist.production` 스펙을 기존 범용적 사양에서 `chrome >= 67`, `and_chr >= 67` 수준으로 낮추어 Create React App(CRA)의 Babel/PostCSS가 ES6 문법 등을 레거시 브라우저 호환 형태로 트랜스파일링하도록 강제했다.
  * 그럼에도 화이트스크린이 해결되지 않아 PC와 TC56 기기를 USB로 연동하여 Chrome DevTools로 원격 디버깅을 시도했고, 콘솔창에서 **`Uncaught ReferenceError: globalThis is not defined`** 런타임 에러를 확인했다.
  * 구형 안드로이드 8.1의 WebView(Chrome 엔진)에는 최신 자바스크립트 표준 전역 객체인 `globalThis`가 구현되어 있지 않아, 앱 번들 및 라이브러리가 로드되자마자 평가 단계에서 뻗어버린 것이 근본적 원인이었다.
* **globalThis 런타임 폴리필(Polyfill) 주입**:
  * 번들 자바스크립트가 실행되기 전에 전역 컨텍스트를 가리키는 `globalThis`를 모방 제공하기 위해, `public/index.html` 최상단 스크립트 영역에 자체 제작한 가벼운 `globalThis` 폴리필 코드를 인라인으로 주입했다.

### Result (결과 및 정량적 지표)
* **구형 기기 구동 성공**: `globalThis` 폴리필 주입 및 빌드 사양 조정 결과, Android 8.1 (TC56) 기기 내의 WebView 런타임에서 화이트스크린 현상이 완전 해결되고 화면이 정상 구동됨을 확인했다.
* **빌드 안정성 확보**: 빌드 Flavor 및 SDK 하향 패치 작업에 따른 런타임 충돌 없이 구버전 기기에서도 안정적인 권한 취득 및 디바이스 관제가 작동한다.

## Takeaway (인사이트)
* **도구 조작 가치**: Google Stitch 웹 제어의 한계를 극복하기 위해 AI에게 MCP라는 강력한 도구 인프라(Schema 접근권)를 마련해 주는 것은 개발 생산성을 크게 바꾼다.
* **기본 명세서의 강력함**: 코드 구현 전 가이드(Do's & Don'ts)를 정형 문서화해 두면 AI가 경계 조건 안에서 동작하여 구현 정합성을 대폭 높일 수 있다.
* **빌드 타깃 하향 vs 런타임 폴리필의 분리**: 구형 브라우저/웹뷰 호환성 대응은 단순히 `browserslist` 같은 빌드 설정 수정만으로 절대 종결되지 않는다. 
  * `browserslist`는 문법적 하향(Syntax Transpiling)만 담당할 뿐, **런타임에 부재하는 내장 전역 객체(API, Global Object)를 자동 생성해주지 못한다.**
  * 전역 API가 누락된 구형 플랫폼 대응 시에는 반드시 `globalThis`처럼 런타임에 누락된 객체를 구현해 주는 **폴리필(Polyfill) 전략이 병행**되어야 근본적인 해결이 가능하다.

# TIL [2026-07-20] - STAR 기법 적용

## Situation & Task (상황 및 문제 정의)
* 미디어텍, 삼성 DS 등 글로벌 칩셋 벤더의 플랫폼/BSP/System Software 엔지니어 직무 지원을 준비하며, 해당 직무의 핵심 기술 키워드 20선과 필수 시스템 개념 8가지(APP~커널 드라이버 E2E 흐름, 콜스택, QEMU, Cuttlefish, HWC HAL, Sync Fence, dmesg, Systrace)를 체계적으로 학습·정리할 필요가 있었음.

## Action (원인 분석 및 해결 과정)
* 키워드 20선을 4개 그룹(OS 하위 레이어 / 디스플레이·그래픽스 / 성능·디버깅 / 하드웨어 제어·검증)으로 분류하고, 지식의 응집도에 따라 knowledge 문서 3개로 재구성.
* 기존 문서와의 중복 방지: Binder/mmap/ION은 `android/ashmem-binder-mmap.md`, HAL/AIDL은 `aaos/hal.md` 등 기존 문서를 교차 참조로 연결하고 신규 내용만 작성.
* 각 개념을 "정의 → 내부 동작 → 벤더 실무 관점(무엇이 최적화/디버깅 포인트인가)" 구조로 기술.

## Result (결과 및 정량적 지표)
* 신규 knowledge 문서 3개 생성 — BSP 코어(12개 개념), 그래픽스 파이프라인(8개 구성 요소, E2E 다이어그램 포함), 디버깅/프로파일링(5개 방법론).
* 칩셋 벤더 직무의 기술 스택 전체가 "앱 → 프레임워크 → HAL → 커널 → 물리 신호"라는 단일 종단 흐름 위에 매핑됨.

## Takeaway (인사이트)
* 디스플레이 파이프라인의 본질은 "버퍼(fd) 전달 + Fence 동기화로 연결된 비동기 파이프라인"이며, 60/120Hz 유지 문제는 결국 각 단계의 프레임 예산(16.6ms/8.3ms) 초과 지점을 Perfetto 타임라인에서 특정하는 문제로 환원됨.
* HWC의 존재 이유는 GPU 합성 부하를 DPU로 위임하는 것 — 벤더 구현의 본질은 자사 DPU의 하드웨어 제약을 HWC 정책으로 노출하는 일.
* 시스템 디버깅의 기본 전략은 레이어 경계(JNI/Binder/syscall) 기준 이분 탐색 — logcat → binder → dmesg 순 하강. 기존 SMU/하네스 검증에서 쓰던 장애 격리 방법론과 동일한 사고 구조.

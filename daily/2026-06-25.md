# TIL [2026-06-25] - STAR 기법 적용

## Situation & Task (상황 및 문제 정의)
* **상황:** 적은 개발 인력 하에 트럭 카운트(Bus Factor) 및 관리/보충 포인트를 최소화하기 위해, 신규 런처 개발 프로젝트의 기술 스택을 React Native(React 생태계)로 통일함.
* **목표:** 개발 생산성 극대화 및 디바이스 전반에 걸친 UI 디자인/동작의 일관성을 확보하고자 함.

## Action (원인 분석 및 해결 과정)
* **설계 접근:** 제품의 안정성을 확보하고 플랫폼 아키텍처에 대한 이해도를 높이기 위해, RN의 3대 핵심 레이어(`Native` - `Bridge` - `React/JS`)를 명확히 구분하고, 각 경계를 교차하는 데이터 흐름을 추적(`TRACE`)할 수 있는 엄격한 플랫폼 아키텍처를 설계함.

## Result (결과 및 정량적 지표)
* **결과 및 난관:** 설계상으로 레이어를 명확히 구분했으나, 오히려 안정성을 담보하기 위한 플랫폼 레이어 개발의 복잡도가 크게 증가함.
* **구체적 현상:** 네이티브 생명 주기(Android/iOS Lifecycle)를 JS 레이어에서 직접 동기화하고 관리해야 했으며, 컴파일러 차원의 강제성(e.g. Kotlin의 `sealed class`)이 약한 환경에서 복잡한 상태를 안전하게 처리하기 위해 상태 머신을 더 엄격히 정의하고 관리해야 하는 예기치 못한 비용이 발생함.

## Takeaway (인사이트)
* React Native는 리소스 제약 상황에서 생산성과 UI 일관성을 확보하는 데 매우 훌륭한 도구이지만, 시스템에 강력히 밀착하여 하드웨어/OS 기능을 직접 제어해야 하는 '런처' 같은 플랫폼 제품의 개발에서는 네이티브-브릿지-JS 경계로 인한 아키텍처적 복잡도가 추가되어 설계 및 상태 관리 난이도가 크게 증가함.
* 플랫폼 단의 안정성을 보장하기 위해서는 단순히 UI 기술 스택을 통일하는 것 이상으로, 런타임 수명 주기와 철저한 상태 격리를 고려한 구조화된 아키텍처 설계가 초기에 반드시 수반되어야 함.

## 공부 메모

### 플랫폼 레이어 개발자의 설계 원칙 (Kotlin)

> 코드보다 판단이 본질 — 코드는 그 판단의 결과물

뛰어난 플랫폼 레이어 개발자는 **정상 동작보다 실패 시나리오를 더 정교하게** 다룬다.

**핵심 5원칙:**

1. **상태 머신 명시화** — `sealed class`로 가능한 상태를 닫고, 전이 함수에서 허용되지 않은 전이는 `Error`로 명시. bool 조합 대신 상태를 하나의 타입으로 표현.

2. **Observability** — 각 스텝을 측정해 타임아웃 수치의 근거를 확보. "LLM 3초 타임아웃"은 실측 p99에서 나와야 함.

3. **백프레셔** — `Channel(capacity=1, DROP_OLDEST)` 또는 `conflate()`로 처리 속도보다 빠른 입력을 방어. IVI에서는 최신 명령만 처리하는 게 맞음.

4. **타임아웃 + 재시도 + 폴백** — 외부 시스템은 반드시 실패한다. `withTimeout` → `retry(2)` → `fallback()` 순서로 설계.

5. **시퀀스 다이어그램으로 팀 소통** — 구현 전에 경계와 실패 경로를 다이어그램으로 합의.

**설계 순서:** 상태 머신 정의 → 지연 측정 → 백프레셔 → 타임아웃/재시도/폴백 → 팀 소통

→ 상세 (Kotlin): `knowledge/platform/platform-design-principles-kotlin.md`

### 플랫폼 레이어 개발자의 설계 원칙 (React Native)

> 네이티브가 공짜로 주는 것을 RN에서는 개발자가 의식적으로 구현해야 한다

**Kotlin 대비 RN의 핵심 차이:**

1. **상태 머신** — `sealed class` + 컴파일 체크 없음 → TypeScript union + `default` 분기로 직접 방어. 복잡해지면 XState 도입.

2. **Observability** — GC/스레드 경합이 아니라 **JS Thread 싱글 스레드 블로킹**이 병목. `InteractionManager.runAfterInteractions()`로 애니메이션 후 무거운 작업 실행.

3. **백프레셔** — `conflate()` 같은 언어 내장 없음 → RxJS `switchMap` 또는 `AbortController`로 직접 구현.

4. **타임아웃/재시도** — `withTimeout` 없음 → `Promise.race()` + `AbortController` 조합. 타임아웃만 걸면 실제 요청은 살아있어 메모리 누수 가능.

5. **레이어 경계** — 플랫폼이 강제하지 않으므로 파일 구조 + 팀 컨벤션으로 명시적 분리 필요.

→ 상세 (RN): `knowledge/platform/platform-design-principles-rn.md`

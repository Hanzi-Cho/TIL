# TIL [2026-07-22] - STAR 기법 적용

## Situation & Task (상황 및 문제 정의)
* **프론트엔드 API 연동 및 고빈도 실시간 데이터 처리 최적화**: `blaster-web` 프로젝트의 모니터링 화면 개정 시, 실시간 상태값, 전압/전류 및 진행률 데이터를 서버로부터 끊김 없이 안전하게 수신하기 위한 API 및 웹소켓(WebSocket/SSE) 배선 구조를 설계하고 구현할 과제 수립.
* **WebSocket 리스너 구독 누수(Leak) 제어**: 싱글톤으로 유지되는 웹소켓 통신 환경에서 화면 전환(마운트/언마운트) 시 리스너 구독 해제가 누락되어 메모리가 누수되거나 콜백이 중복 실행되어 UI가 오동작하는 에러를 선제적으로 방지하기 위한 가버넌스 및 설계 확립 필요.

## Action (원인 분석 및 해결 과정)
* **4계층 구조 기반 API 아키텍처 및 SSOT 인터페이스 수립**:
  - `ApiConstant.ts`(엔드포인트 경로) $\rightarrow$ `HttpClient.ts`(Axios 공통 처리) $\rightarrow$ `*Service.ts`(순수 응답 데이터 추출) $\rightarrow$ `hooks/*.ts`(상태/실시간 결합 훅) 구조로 단방향 책임을 분리하고 컴포넌트의 API 직접 호출을 철저히 배제함.
* **REST 스냅샷 및 고빈도 실시간 스트림(WebSocket/SSE) 하이브리드 합성**:
  - 화면 마운트 시 최초 1회만 대용량 REST 스냅샷 데이터를 조회하고, 이후 발생하는 변경사항은 WebSocket/SSE 실시간 이벤트로만 수신.
  - 고빈도 데이터(전압, 전류 등)가 수신될 때마다 React State를 직접 갱신하는 대신 `useRef` 버퍼 공간에 이벤트를 누적하여 렌더링 스로틀링을 확보하고, 도메인 순수 함수(`deriveMonitoringPhase()`, `buildCellDetonatorStats()`)를 통해 배치로 통계를 재계산한 후 `setState`를 실행해 렌더링 성능을 개선함.
* **수명주기 한정적 리스너 임시 등록 및 자동 소거 기법 설계**:
  - `useWebSocketConfig` 훅을 구현하여 컴포넌트 생명주기에 맞게 고유 ID를 부여한 임시 리스너(`registerTemporaryListener`)를 달고, Cleanup 시점에 해당 ID로 완벽히 리스너를 소거(`unregisterTemporaryListener`)하도록 표준 코딩 패턴 정의.
* **네트워크 아키텍처의 개념화 및 일상 비유 모델 구축**:
  - REST GET(요청 즉시 답 듣고 끊는 단발성 전화), SSE(라디오 청취 같은 서버-클라이언트 단방향 푸시), WebSocket(항상 열려 있는 전화 통화 같은 양방향 커넥션) 개념을 전화 통화 모델로 비유하여 명확화.
  - Kotlin Flow의 `collect()` 연산 및 IoT MQTT의 Topic `subscribe()` 개념과의 기능적 공통성 및 물리적 프로토콜 차이 비교 분석.
* **코드베이스 실증 감사 및 프로젝트 관례 정합성 분석 (Rule 0)**:
  - 사양 분석 프롬프트를 설계하여 Claude Code로 현재 소스 코드를 전수 감사함. 그 결과 기존 문서에 있던 SSE `/monitoring/events` 호출은 실제 라우트 상에서 미사용되는 사멸 코드(Dead Code)임을 식별하고, 실시간 모니터링은 오직 WebSocket만 사용하고 있음을 실증함.
  - React Query의 교과서적 사용법인 `useQuery` 대신, 단발 스냅샷 패치 및 소켓 동기화 시나리오를 다루는 프로젝트 고유의 훅 관례(예: `useArmRuntime.ts`가 사용하는 `useMutation.mutateAsync` 호출 방식)를 발굴하여 모니터링 API 훅의 리팩토링 설계 방향성을 프로젝트 표준과 일치시킴.

## Result (결과 및 정량적 지표)
* `blaster-web` 프로젝트의 4계층 API 호출 규격 및 실시간 스트림 합성 구조를 완전 파악하고 정리함.
* [api-wiring-pattern.md](../knowledge/frontend/api-wiring-pattern.md) 신규 기술 문서를 생성 및 갱신하여 프론트엔드 API 통신 및 실시간 합성, 리스너 누수 방지 설계 표준 구축 완료.
* 실제 프로덕션 코드 실행 경로 정적 추적을 통해 미사용 사멸 코드(Dead Code) 및 스펙 드리프트를 선제 식별함.
* 프로젝트 고유의 `useMutation.mutateAsync` 비동기 호출 패턴을 적용하여, 실시간 웹소켓 이벤트 렌더링 충돌율 0% 및 소스 일관성을 보존함.

## Takeaway (인사이트)
* API 연결 작업을 처음 접할 때 가장 겪기 쉬운 실수인 '무분별한 Axios 호출'과 'WebSocket 구독 누수'를 막기 위해서는 명확한 계층 분리(4-Tier)와 컴포넌트 수명주기에 연동된 리스너 등록/해제 가버넌스를 설계하는 것이 핵심임을 깊이 깨달음.
* 통신 채널의 방향성(단방향/양방향)과 요청 주도권(Pull/Push)을 구분하여 일상적 비유(전화, 라디오)로 모델링함으로써 복잡한 비동기 이벤트 스트림 코드를 직관적으로 읽고 작성할 수 있게 됨.
* **"문서나 범용 아키텍처 이론보다 실제 살아 움직이는 코드베이스(Truth on Codebase)가 우선한다"**는 교훈을 얻음. 코드를 수정하기 전 실행 경로를 역추적(Rule 0)하고 프로젝트 고유 관례(Project Idiom)를 준수하는 것이 파편화 없는 건강한 리팩토링의 핵심 성공 요인임.

# TIL [2026-06-23] - Design System 구축 파이프라인 정리

## Situation & Task (상황 및 문제 정의)
* 오늘은 모니터링 기능 화면을 단순 화면 수정이 아니라 `문서 -> 디자인 시스템 -> 프론트엔드 코드`로 이어지는 구조로 바꾸기 위한 초석을 정리했다.
* Claude를 활용해 AS-IS / TO-BE 요구사항, 버그 리포트, 화면 흐름, 화면별 구성요소를 PPT 문서로 정리했다.
* 그 다음 단계로 Google Stitch에서 Figma 수준의 디자인 시스템을 만드는 작업을 진행 중이다.
* 최종 목표는 예쁜 시안 하나를 만드는 것이 아니라, 화면 규칙과 상태 표현을 텍스트와 디자인 시스템으로 명확히 규정한 뒤 더 높은 품질의 프론트엔드 코드 설계와 리팩토링으로 연결하는 것이다.

## Action (원인 분석 및 해결 과정)
* 먼저 디자인 시스템을 "공통 컴포넌트 모음"이 아니라 `의사결정의 기록 + 재사용 가능한 토큰 + 구현 가이드`로 봐야 한다는 관점을 정리했다.
* 모니터링 기능 재설계 요구사항을 화면 단위로 구조화했다.
* 상태 체계를 복잡한 다색 상태에서 `Stage 3단계 + State 4색`으로 단순화했다.
  * Stage: `COMMUNICATION -> ARM -> COMPLETED`
  * State color: 시작 회색, 진행 중 노란색 점멸, 완료 초록색, abort/에러 빨간색
* 화면 구조 설계 시 반드시 고정해야 할 레이어를 먼저 분리했다.
  * 좌측 바는 구성뿐 아니라 간격과 디자인까지 수정 금지
  * 상단 주황색 `PREV` 버튼도 수정 금지
* 그룹 보드는 3안을 채택하는 쪽으로 정리했다.
  * 그룹 간 이동은 `Prev` / `Next`
  * 같은 그룹에서 디바이스가 5개 이상이면 내부 스크롤
  * 기본 가시성은 2x2 카드 뷰 유지
  * 다만 4개만 보여서 5개 이상 존재를 놓칠 수 있으므로 overflow 존재를 알려주는 신호 설계가 필수
* 우측 상세 정보는 확장 탭이 아니라 상시 노출 패널로 유지하고, 좌측 남는 공간에 2x2 디바이스 카드를 배치하는 구조를 유지하기로 판단했다.
* 데이터 중복 제거 규칙도 같이 정리했다.
  * 그룹 우하단 `NG` 요약 제거
  * `delay time`은 state 열의 우측 끝으로 이동
  * 상세 패널에서 `error detonators` 옆 `NG count` 제거
  * 화면 내 텍스트는 모두 영어로 통일

## Result (결과 및 정량적 지표)
* 단순한 UI 수정 요청이 아니라, 다음처럼 재사용 가능한 파이프라인으로 사고를 전환했다.
  * `PPT screen spec -> Stitch design system -> token/component guideline -> frontend refactor`
* 화면 설계 시 앞으로 반드시 먼저 정리해야 할 체크포인트가 생겼다.
  * 어떤 영역이 절대 수정 금지인지
  * 상태 전이를 Stage와 State로 분리했는지
  * 컴포넌트가 중복 데이터를 보여주지 않는지
  * 실제 사용 해상도에서 가시성이 충분한지
* 프론트엔드 입문자 관점에서도 "표기 로직이 복잡해지는가?"를 막연히 느끼는 대신, `그룹 간 페이징`과 `그룹 내부 스크롤`을 책임 분리된 UI 규칙으로 이해하게 되었다.

## Takeaway (인사이트)
* 디자인 시스템은 시안을 예쁘게 만드는 도구가 아니라, 나중에 프론트엔드 코드가 흔들리지 않게 하는 결정 체계다.
* 특히 이번 작업에서 가장 중요했던 것은 "화면을 어떻게 그릴까?"보다 "어떤 것은 절대 안 바꿔야 하는가?"를 먼저 고정한 점이다.
* 좋은 파일 구조는 기술 이름 기준보다 의사결정 흐름 기준으로 나누는 편이 낫다.
  * 예: `tokens`, `components`, `patterns`, `screens`, `prompts`, `scripts`
* 파일 구조를 설계할 때는 다음 질문이 유효했다.
  * 이 파일은 사람이 읽기 위한가, 도구가 읽기 위한가
  * 화면 요구사항이 토큰과 컴포넌트까지 추적 가능한가
  * Stitch 입력, 코드 생성, 리팩토링 검수까지 이어지는가
* 앞으로 디자인 시스템 문서를 만들 때 필수로 고려해야 할 것들:
  * Token first: 하드코딩 값보다 토큰을 먼저 정의할 것
  * Semantic naming: `red-500`보다 `danger`, `progress`, `surface-panel`처럼 의미 기준으로 이름 지을 것
  * Fixed-area rules: 수정 금지 영역을 문서 최상단에 둘 것
  * Accessibility: 색만으로 상태를 전달하지 말 것
  * Traceability: 화면 요소가 어느 컴포넌트와 토큰에 매핑되는지 남길 것
  * Automation-ready: 프롬프트와 스크립트가 재사용 가능하도록 구조화할 것

# TIL [2026-06-23] - TIL README 학습 통계 자동화 구축

## Situation & Task (상황 및 문제 정의)

* TIL 레포의 README 상단에 도메인별 학습량 통계(최근 3개월 / 전체 누적)를 자동으로 표시하는 시스템이 필요했다.
* 커밋할 때마다 또는 매일 자정에 git 히스토리를 분석해 README를 갱신하는 파이프라인을 만들어야 했다.

## Action (원인 분석 및 해결 과정)

* `scripts/update_readme.py` 작성
  * `git log --name-only`로 커밋별 변경 파일을 파싱
  * `knowledge/<domain>/` 경로 파일만 집계 대상으로 삼아 `daily/`, `scripts/` 등을 자동 제외
  * `os.listdir("knowledge/")` 로 서브디렉토리를 런타임에 탐색 → 새 도메인 추가 시 스크립트 수정 불필요
  * 비중 = 해당 도메인 커밋 수 / 전체 도메인 커밋 수 합계 (합산 100%)
  * KST(UTC+9) 타임존을 `timezone(timedelta(hours=9))`로 명시
  * quickchart.io로 도넛 차트 URL 생성, geps.dev로 진행 바 이미지 생성

* `.github/workflows/update-readme.yml` 작성
  * `push` (main 브랜치, README.md 제외) + `schedule: '0 15 * * *'` (KST 자정) 트리거
  * `fetch-depth: 0` 으로 전체 커밋 히스토리 가져오기
  * git config를 본인 계정 이메일로 설정 → 자동 커밋도 기여 그래프(잔디)에 반영
  * `[skip ci]` 커밋 메시지 + `paths-ignore: README.md` 이중 무한루프 방지

* README 레이아웃 설계
  * HTML `<table>`로 도넛 차트와 랭킹 테이블을 좌우 2단 배치
  * GitHub HTML sanitizer가 `<td width>` 를 제거하므로 이미지 너비 제어는 `<img width>` 태그로만 가능함을 확인

## Result (결과 및 정량적 지표)

* 매일 KST 00:00 또는 main 브랜치 push 시 README 상단 통계가 자동 갱신된다.
* `knowledge/` 아래 새 디렉토리를 만들면 다음 실행부터 자동으로 집계에 포함된다.
* 자동 커밋이 `Hanzi-Cho <whwogus1983@gmail.com>` 명의로 기록되어 GitHub 잔디가 깔린다.

## Takeaway (인사이트)

* **GitHub Actions cron은 UTC 기준**이다. KST 자정(00:00)은 `0 15 * * *`.
* **GitHub HTML sanitizer는 `<td width>`를 제거**한다. README에서 HTML 테이블 컬럼 너비를 직접 제어할 수 없으며, 이미지라면 `<img width="N">` 태그를 써야 한다.
* **무한루프 방지**는 `paths-ignore`와 `[skip ci]`를 함께 쓰는 것이 안전하다.
* 자동화 스크립트가 지속적으로 가치를 내려면 "새 항목 추가 시 스크립트를 수정하지 않아도 되는 구조"를 처음부터 설계해야 한다.

# TIL [2026-06-23] - 디자인 시스템 심화 & Google Stitch 실전 한계

## Situation & Task (상황 및 문제 정의)

* 모니터링 화면 재설계를 진행하면서 디자인 시스템을 단순 컴포넌트 모음이 아닌 `의사결정 체계`로 구조화하는 작업을 했다.
* Google Stitch를 실제 프로젝트에 투입하면서 할 수 있는 것과 없는 것의 경계를 직접 부딪혀 확인했다.
* Figma와 Stitch의 역할을 어떻게 분리해야 하는지 정리가 필요했다.

## Action (원인 분석 및 해결 과정)

### 디자인 시스템 6계층 정리

단순 UI 파일이 아닌 6개 레이어로 구성되는 구조를 정리했다.

```
L1  Context       CLAUDE.md — AI에게 도메인 규칙 주입
L2  Guardrail     수정 금지 규칙 자동 강제 (pre-commit, CODEOWNERS)
L3  Token         색상·간격·타이포 단일 소스 (코드가 SSoT)
L4  Component     공통 컴포넌트 / 도메인 컴포넌트 분리
L5  Validation    Static → Unit → Integration → E2E 순서
L6  History       ADR · PR · Figma 버전 히스토리
```

**공통 vs 도메인 컴포넌트 분리 원칙**

```
공통                          도메인
────────────────              ────────────────────────
Button, Badge, Dialog         StateChip, StageStamp
tokens/base.ts                tokens/state-colors.ts
도메인을 모름                  State enum 직접 참조
props로 색상 주입받음           4색 토큰만 사용
```

**가드레일 기록 위치**

| 종류 | 위치 | 이유 |
|---|---|---|
| 수정 금지 규칙 | `CLAUDE.md` + `CODEOWNERS` | AI·사람 동시 차단 |
| 검증 명세 | 테스트 파일 (`__tests__/`) | 실행 가능한 명세 |
| 결정 이유 | `docs/adr/` ADR | "왜"를 코드로 알 수 없음 |
| 시각적 히스토리 | Figma 버전 히스토리 | before/after 필요 |

**실무 전달 도구별 역할**

| 도구 | 역할 |
|---|---|
| Figma | 레이아웃·dp·색상·버전 히스토리·Annotation |
| Notion | 규칙 문서·변경 로그·기획 명세서 |
| Jira / Linear | 버그 리포트·이슈 추적·PR 연결 |
| Storybook | 컴포넌트 가이드 (개발 주도) |
| Slack | 비동기 리뷰 → 결정은 Notion으로 이관 |

**검증 파이프라인 우선순위 (1인 개발 기준)**

```
1주차  CLAUDE.md + State machine 단위 테스트 + pre-commit hook
2주차  MSW 네트워크 에러 + 대량 디바이스(0/1/100/500/1000)
3주차~ Playwright E2E (위 레이어 안정 후)
```

### Google Stitch 실전 한계 확인

**할 수 있는 것 / 못하는 것**

```
✅ 텍스트 프롬프트 → UI 화면 생성
✅ 스크린샷 첨부 → 레이아웃 재현
✅ 컴포넌트 Variant 생성 (State별 변형)
✅ 여러 장 동시 첨부 → variant 비교 생성

❌ 프레임 px 크기 강제 (2000×1200 지시 무시)
❌ 기존 프레임 삭제·초기화
❌ 캔버스 그리드 배치
❌ 화면 간 화살표·연결선 생성
❌ 콘텐츠 많으면 세로로 자동 확장 (비율 무시)
```

**Stitch 프롬프트 원칙**

```
영어로 작성     지시어·수치·제약은 영어가 정확도 높음
한글은 유지     실제 UI에 표시되는 텍스트는 한글 그대로
짧고 명확하게   긴 설명보다 수치·색상 hex 명시가 효과적
변경 요소만 명시  "Based on previous screen, change only:"
이미지 동시 첨부  variant 생성 시 여러 장 한번에 주기
```

**Figma 선 작업 → Stitch 후 작업 워크플로우**

```
Figma에서 2000×1200 프레임 이름 지정
        ↓
Stitch 프롬프트 앞에 항상 추가:
"Draw inside the existing Figma frame named {이름}.
 Do not create a new frame.
 Fill the entire 2000×1200 boundary."
        ↓
Stitch 결과 → Figma로 export
        ↓
Figma에서 배치·화살표·정렬 완성
```

## Result (결과 및 정량적 지표)

* 디자인 시스템을 6계층으로 구조화해 각 레이어의 책임을 명확히 분리했다.
* Stitch는 "개별 화면 스타일 생성" 전담, Figma는 "px 배치·연결·정렬" 전담으로 역할을 분리해 워크플로우를 확립했다.
* 코드(TypeScript token)를 SSoT로 삼고 CLAUDE.md는 참조 문서로만 쓰는 원칙을 확립했다.

## Takeaway (인사이트)

* **코드 SSoT 없이 MD 파일만 있으면 규칙이 반드시 drift된다.** 코드와 문서 둘 다 규칙을 가지면 코드가 이긴다.
* **Stitch는 "디자인 생성 도구"지 "캔버스 제어 도구"가 아니다.** px 크기·배치·삭제는 Figma에서 해야 한다.
* **프롬프트가 길수록 세로로 늘어난다.** 섹션을 분리해서 짧게 입력하는 것이 결과 품질에 유리하다.
* **Figma Annotation이 없으면 도메인 규칙이 화면에서 사라진다.** 수정 금지 영역, 상태 정의 등은 반드시 Annotation으로 고정해야 한다.
* 검증 파이프라인은 위에서 아래로 쌓는 게 아니라 **안정된 레이어부터 순서대로** 쌓아야 실용적이다.

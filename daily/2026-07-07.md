# TIL [2026-07-07] - STAR 기법 적용

## Situation & Task (상황 및 문제 정의)
* 리액트의 근원적 아키텍처를 복습하던 도중, 함수형 컴포넌트가 매번 소멸되고 재실행되는 과정에서도 상태(`useState`)가 유실되지 않고 유지되는 물리적 저장소에 대한 이론적 공백을 인지함.
* 클래스 컴포넌트 관점에서의 보존 영역인 '컴포넌트 인스턴스(Instance)'와 대비하여, 모던 리액트의 상태 보존 메커니즘인 **파이버 노드(Fiber Node)** 및 **파이버 아키텍처(Fiber Architecture)**의 개념과 동작 원리를 규명해야 하는 과제를 설정함.
* **[추가 태스크]** `useEffect` 훅의 핵심 메커니즘인 **클린업 함수(Cleanup Function)**의 실행 조건 및 **의존성 배열(deps)** 유무에 따른 동작 방식 차이, 그리고 렌더링 라이프사이클 속의 상세 실행 순서를 파악해 정비함.
* **[추가 태스크 2]** 리액트 컴포넌트의 상태 전파와 가상 DOM 반영 파이프라인의 핵심인 **세부 렌더링 라이프사이클 타임라인(Trigger ➔ Render ➔ Commit ➔ Paint ➔ Passive Effects)**을 수립하여 수명 주기를 정밀하게 구조화함.
* **[추가 태스크 3]** 리액트 공식 빌드 아키텍처에서 기본 제공하는 **15종의 내장 Hook 전체 명세**를 용도와 성격별(상태, 이펙트, 컨텍스트, 메모이제이션, 동시성 유틸)로 분류하여 구조화함.
* **[추가 태스크 4]** 상태 변경 함수(`setState`)의 비동기적 예약 특성, 컴포넌트 스냅샷에 의한 동치 예약 오류 및 이를 해결하기 위한 **배치 업데이트(Batching/Automatic Batching)**와 **함수형 업데이트**의 동작 기작을 명확히 정리함.
* **[추가 태스크 5]** 안드로이드 OS 플랫폼의 동작 사상과 시스템 아키텍처 복습을 위해 **4대 핵심 컴포넌트**의 정의 및 실제 하드웨어/BLE 연동 연계 시나리오, 액티비티 생명주기(onPause/onStop의 LMK 관점 구분), 그리고 명시적/암시적 Intent의 차이를 정리함.

## Action (원인 분석 및 해결 과정)
* **Stack Reconciler의 한계 분석**:
  * 리액트 15 이전의 동기식 재조정(Stack Reconciler)은 자바스크립트의 Call Stack을 독점하여 렌더링 중단을 방해함으로써, 거대 트리 구조에서 화면이 버벅이는 Jank 현상을 유발했음을 파악함.
* **Fiber Node 및 가상 스택 프레임 규명**:
  * 각 컴포넌트의 상태 관리를 위해 1:1로 매핑되는 **Fiber Node**가 메모리에 유지됨을 이해함.
  * `useState` 등의 훅 상태는 이 파이버 노드의 `memoizedState` 필드 내에 **단방향 연결 리스트(Linked List)** 형태로 누적 관리되어 함수 재호출 시에도 보존됨을 정리함.
  * 작업을 조각으로 쪼개어 중단(Pause), 재개(Resume), 폐기(Abort)하는 **동시성 렌더링(Concurrent Rendering)**을 위한 스케줄링 메커니즘을 파악함.
* **더블 버퍼링(Double Buffering) 구조 정리**:
  * 메모리 내에 브라우저 실물 화면을 뜻하는 `current` 트리와 상태 갱신 연산 중인 `workInProgress` 트리를 이중으로 구성하고, 연산 종료 후 포인터만 교체하여 렌더링 깜빡임을 없애는 더블 버퍼링 아키텍처를 분석함.
* **useEffect 클린업 및 의존성 메커니즘 교정**:
  * **클린업의 정의와 발생 조건**: 클린업(return하는 함수)은 메모리 누수를 막는 리소스 해제 작업이며, ① 컴포넌트가 사라지는 언마운트(Unmount) 시점뿐 아니라 ② **의존성 배열(deps) 값 변경으로 인해 새로운 Effect가 실행되기 직전**에도 항상 먼저 실행되어 이전 흔적을 정리함을 정립함.
  * **deps 인자 유무에 따른 치명적 차이 교정**: 의존성 배열을 생략한 `useEffect(fn)`는 **리렌더링될 때마다 매번 실행**되는 반면, 빈 배열 `[]`을 명시한 `useEffect(fn, [])`는 **마운트 시 최초 1회만 실행**되는 차이점을 완벽하게 구분함.
  * **렌더링 수명주기 5단계 정밀화**:
    1. **Trigger (렌더링 요청)**: `setState` 호출, 부모 컴포넌트의 리렌더링, 혹은 Context 변경으로 시작함.
    2. **Render Phase**: 가상 DOM 트리 생성 및 휴리스틱 Diffing 비교(Reconciliation) 수행.
    3. **Commit Phase**: DOM 반영 단계로, ① Before Mutation(스크롤 위치 캡쳐 등) ➔ ② Mutation(실제 DOM 노드 수정) ➔ ③ Layout Phase(`ref` 값 업데이트, 이전 `useLayoutEffect` 클린업 및 새 `useLayoutEffect` 즉시 실행) 순으로 동기 처리됨을 매핑함.
    4. **Paint**: 브라우저 엔진이 레이아웃을 다시 잡고 픽셀을 입혀 그리는 과정.
    5. **Passive Effects**: 브라우저 페인팅 후, 비동기적으로 이전 `useEffect` 클린업을 먼저 돌린 후 새 `useEffect`를 호출함.
* **내장 Hooks 분류 체계 수립**:
  * 리액트의 내장 Hooks 15종을 상태(`useState`/`useReducer`/`useRef`), 이펙트(`useEffect`/`useLayoutEffect`/`useInsertionEffect`), 컨텍스트(`useContext`), 메모이제이션(`useMemo`/`useCallback`), 동시성 및 유틸(`useId`/`useDeferredValue`/`useTransition`/`useSyncExternalStore`/`useImperativeHandle`/`useDebugValue`)로 일목요연하게 계층화하여 비교 기술함.
* **setState 비동기 메커니즘과 배치 업데이트 분석**:
  * **setState 예약성 동작 규명**: `setCount(count + 1)`을 실행해도 해당 렌더링 컨텍스트에서는 `count`가 불변의 스냅샷 상수이므로 즉시 바뀌지 않고 다음 렌더링을 예약하는 성질을 파악함.
  * **Batching & Automatic Batching 분석**: 다중 상태 업데이트를 묶어 한 번에 렌더링하는 배치 기법과, React 18부터 Promise/setTimeout 콜백 영역까지 전방위로 자동 배칭이 적용되는 **Automatic Batching**의 메커니즘을 정립함.
  * **함수형 업데이트 적용 조건 도출**: 연속 호출 시 동치 예약으로 인한 값 누락 버그를 예방하기 위해, 이전 예약 임시 값을 인자로 연결하는 **함수형 업데이트(`prev => prev + 1`)**의 동작 구조를 이해함.
* **안드로이드 핵심 플랫폼 아키텍처 분석**:
  * **4대 컴포넌트의 실무 가치**: 액티비티(Activity)의 SAMF 구조 활용, 백그라운드 BLE 및 하드웨어 스캔 연동 시 Service의 작업 분리 개념을 정립하고, Broadcast Receiver와 Content Provider의 기본 개념을 정비함.
  * **액티비티 생명주기 및 LMK 임계 구분**: 다이얼로그나 팝업 등으로 일부 가려진 **`onPause()`**와 아예 백그라운드로 전환되어 화면에 보이지 않는 **`onStop()`**의 물리적 시각 경계를 수립함. 특히 `onStop()` 상태에서 시스템 메모리가 부족할 경우 **Low Memory Killer (LMK)**에 의해 강제로 프로세스가 해제될 수 있는 저레벨 연계점을 구체화함.
  * **Intent의 두 가지 용도 격리**: 대상 목적지를 명확히 박아서 호출하여 앱 내부 이동에 쓰이는 **명시적 Intent**와, 특정 액션과 미디어를 선언해 시스템에 처리를 위임하는 외부 앱 연동 목적의 **암시적 Intent**의 역할을 대조 정립함.

## Result (결과 및 정량적 지표)
* 가상 DOM과 리액트의 재조정자 설계 철학 및 훅스(Hooks) 라이프사이클을 완벽히 교정하고, 지식 문서인 [react-concept.md](../knowledge/frontend/react-concept.md)에 관련 항목들을 추가함.
* 추가로 안드로이드의 저레벨 아키텍처를 다룬 [core-concepts.md](../knowledge/android/core-concepts.md) 지식 문서를 신설 확보하여 모바일 포트폴리오 면접 통과 수준의 기반 기술 학습 체계를 구축함.

## Takeaway (인사이트)
* 리액트 함수형 컴포넌트는 순수 함수(Pure Function)로 설계되었지만, 그 내부 훅의 생명주기는 React 엔진이 가상 DOM(Fiber Tree) 메모리에 보존해 주는 가상 Stack Frame(Fiber Node)의 포인터 체인에 강하게 기탁하여 작동함을 배움.
* `useEffect` 의존성 배열의 인자 생략과 빈 배열 `[]` 간의 사소해 보이는 코드 차이가 렌더링 루프 폭발을 유발하거나 메모리 누수를 발생시키는 중대한 원인이 됨을 인지하고, 라이프사이클 Paint 시점 제어의 중요성을 학습함.
* 모바일 애플리케이션의 핵심 라이프사이클 흐름(onResume/onPause/onStop)과 Intent 전달은 웹의 라우팅 및 렌더링/비동기 예약 아키텍처와 매우 유사한 철학(SAMF 아키텍처 등)을 공유하고 있음을 발견하여, 크로스 플랫폼 관점의 아키텍처 혜안을 기르게 됨.
